## 一、项目定位

多智能体团队协作平台，用户通过主智能体（项目经理角色）调度不同 AI 智能体协作完成复杂任务。面向非程序员个人用户，界面类似微信聊天框。

**技术栈**：Python 3.10 + FastAPI（单文件 `backend/main.py`）+ Vue 3 + Pinia + Element Plus + Vite；纯 JSON 文件存储（无数据库）；本地单机部署。

**明确不做的边界**：手机端、用户登录、云端部署、多用户协作、复杂权限继承。不要主动实现这些。

## 四、关键架构概念

| 概念 | 要点 |
|---|---|
| 项目=对话 | `conv_id` 以 `proj_` 开头，项目管理走 `/api/conversations*`，不是两套系统 |
| 智能体三级配置 | 全局模板(`data.json`) → 团队级(`teams.json`) → 项目级(`conversations.json`)；修改全局模板不影响已有项目 |
| 智能体分级 L0/L1/L2 | L0 主智能体（全权限）/ L1 子智能体（按模板）/ L2 执行者（单任务） |
| 五个通用模板 | `creator`/`reviewer`/`analyst`/`worker`/`coordinator`，与领域正交 |
| 调度模式 | 直接模式、链式调度（max_rounds=6）、并行调度（MAX_PARALLEL=5）、协调者两级调度、临时子代理 |
| 三大扩展系统 | 技能系统、蓝图系统、元团队（v0.4.0 已重构为常驻专家+三阶段引擎+评审闭环） |

**元团队架构**（v0.4.0 已重构）：不再是"项目类型"或"固定四成员"，而是独立的常驻专家系统。默认 3 个常驻专家，deep 模式三阶段（方案撰写→评审→融合），fast 模式单专家直接产出。详见 [元团队重构方案](docs/04-设计/20260720-元团队常驻化与工作模式重构方案.md)。

## 五、权限隔离框架（核心安全机制，勿动）

项目运行时目录结构：
```
项目根目录/                      ← 用户设置的工作目录（root_path）
├── .agent/tasks/{task_id}/      ← 任务隔离区（核心）
│   ├── input/                   ← 主智能体放的输入文件（子智能体只读）
│   └── output/                  ← 子智能体的产出（子智能体可写）
├── agent_work/{agent_id}/       ← 智能体私有工作区（互不可见）
├── shared/                      ← 共享文件区（所有智能体可读）
└── deliverables/                ← 最终交付物（主智能体归档）
```

**红线**：
- `.agent/tasks/` 是权限隔离核心，每个任务独立空间，不要破坏
- 子智能体产出是中间文档，不是最终交付，主智能体负责用 `move_file` 归档到 `deliverables/`
- **不要**让子智能体直接写 `deliverables/`，会破坏权限隔离
- **不要**让子智能体写入 `input/` 目录，后端会拒绝

## 六、防幻觉机制（四层防御，保持完整）

1. **提示词硬约束**：`CORE_RULES` 真实性约束，自动注入所有智能体
2. **归档动作回读校验**：`move_file`/`copy_file` 执行后自动回读目标文件，返回字节数证据
3. **任务提交校验**：`submit_task_result` 严格校验 task_id + 归属 + pending 状态 + 结果文件真实性
4. **调度循环幻觉检测**：扫描主智能体回复中的 `task_xxx` 引用，校验不存在则发 `hallucination_warning` 事件

任何一层失效都可能导致智能体编造结果，修改时保持完整。

## 七、前后端接口约定

### 返回格式

| 类型 | 格式 | 注意 |
|---|---|---|
| 列表接口 | `{"资源复数名": [...]}` | 如 `{"conversations": [...]}`、`{"agents": [...]}` |
| 文件列表 | `{"items": [...], "path"}` | ⚠️ 用 `items` 不是 `files` |
| 文件对象 | `{"name", "is_dir", "path"}` | ⚠️ 用 `is_dir`(bool) 不是 `type` |
| 成功操作 | `{"status": "ok"}` | 通用成功标记 |
| 错误 | `HTTPException(4xx, detail)` → `{"detail": "..."}` | 没有 `{"success": false}` 格式 |

**前后端字段必须同步**：改后端返回格式时，必须同步改前端解析。前端不判断 `res.data.success`。

### SSE 事件类型
`agent_start`、`token`、`tool_start`、`tool_end`、`agent_end`、`batch_start`、`batch_end`、`error`、`done`、`hallucination_warning`

## 八、修改红线

### 必须遵守
1. **先读相关代码再改**——用 Read 工具看完整上下文，不要凭记忆改；用函数名搜索定位，不依赖行号
2. **保持权限隔离完整**——`.agent/tasks/` 结构、`allowed_paths` 白名单、`check_agent_path_access` 校验
3. **保持防幻觉机制完整**——四层防御不能少
4. **前后端字段同步**——改返回格式必须同步改解析
5. **数组更新用 splice**——Vue3 直接索引赋值不触发响应式更新
6. **API 密钥脱敏处理**——后端返回脱敏值，前端编辑保存时若检测到脱敏值（含 `***`）必须删除 payload 中的 api_key 字段
7. **工具定义与执行成对修改**——新增工具时：`TOOL_DEFINITIONS` 加定义 + `execute_tool` 加执行逻辑 + 按需在 `MAIN_ONLY_TOOLS`/`SUB_AGENT_TOOLS` 登记 + 更新 `CORE_RULES` 说明
8. **文件操作用专用工具**——禁止用 Shell 命令（cat/sed/rm/find/grep 等）操作文件，用 Read/Edit/Write/DeleteFile/Glob/Grep
9. **每次只改一个点**——验证后再改下一个
10. **通义千问容错不要移除**——`function.arguments` 四层容错解析，移除会导致 400 错误

### 代码风格
- 后端：函数 `snake_case`，Pydantic 模型 `PascalCase`
- 前端：组件 `PascalCase.vue`，函数 `camelCase`
- 注释用中文

## 九、大模型调用约定

- **配置优先级**：智能体 `model_config.config_id` > 会话级 `default_config_id` > 全局默认 `load_api_config()`
- **上下文窗口**：`context_window` 可配（4096~1048576，默认 131072），截断阈值 = context_window - 8000(输出) - 2000(安全)，最小 30000
- **上下文隔离**：子智能体不继承全部历史，只接收任务描述 + input 指引（`build_isolated_context`）
- **思考模式**：`enable_thinking` 通过 `extra_body` 传入，部分服务商不支持，**必须保留自动回退机制**

## 十、数据存储约定

纯 JSON 文件存储，无数据库。关键文件：

| 文件 | 用途 | 红线 |
|---|---|---|
| `data.json` | 全局智能体模板（仅 main） | **不要**写入子智能体，会污染所有新项目 |
| `configs.json` | 多模型配置库 | api_key 明文存储，前端展示需脱敏 |
| `conversations.json` | 项目级智能体配置 | `create_team_member` 只写这里 |
| `teams.json` | 团队级智能体配置 | 团队是项目容器，新建项目可继承 |
| `messages.json` | 所有项目聊天消息 | 全局共享，按 conversation_id 区分 |

`/api/agents` 接口每次调用 `load_agents()` 重新加载，不使用缓存。

## 十一、新增功能指引

### 新增 API 接口
1. 后端 `main.py` 加 `@app.post("/api/xxx")` → 2. 返回格式遵循第七节约定 → 3. 前端 `api/index.js` 加方法 → 4. Store action 调用

### 新增智能体工具
1. `TOOL_DEFINITIONS` 加定义 → 2. `execute_tool` 加执行逻辑 → 3. `MAIN_ONLY_TOOLS`/`SUB_AGENT_TOOLS` 登记 → 4. 更新 `CORE_RULES` 说明 → 5. 前端 `handleSSEEvent` 的 `tool_end` 处理特殊展示

> 用函数名搜索定位上述位置，不要依赖行号。

### 工具分配规则
- **主智能体**：工具过滤层可用全部 `TOOL_DEFINITIONS`，但 `create_sub_task` 和 `invoke_sub_agent` 在执行层有额外校验
- **子智能体**：按 `template` 对应 `ROLE_TEMPLATES` 静态工具列表过滤，再合并 `enabled_skills` 技能工具
- **临时子代理**：强制 `can_invoke_sub_agent=False`（非递归）、`enabled_skills=[]`（不继承技能）

## 十二、过程文档维护规则

> 本节来自用户工作偏好，所有接手本项目的 AI 智能体必须遵守。详见 [文档维护规范](docs/03-指导/20260719-文档维护规范.md)。

**触发门槛**：新增/修改/删除代码文件、API 接口、工具定义、数据结构，或修复 bug、新增功能模块时触发。纯问答/讨论不触发。

**有产出时必做**：
1. 追加会话纪要到 `docs/06-变更日志/YYYYMMDD-会话纪要.md`（当天有则追加，用 `---` 分隔）
2. 追加变更日志到 `docs/CHANGELOG.md`（顶部追加，格式：`| 日期 | 变更类型 | 一句话描述 |`）

**单一权威来源**：每个知识点只能有一个权威文档，其他文档引用不复制。权威源对照表维护在 `docs/README.md`。新建文档前先搜索是否已有同类文档，有则更新不新建。AGENTS.md 只写关键约束，细节指向对应权威文档。

**文档命名规范**：活文档（长期维护、反映当前状态，如代码架构说明、功能范围清单）用语义化命名，不带日期前缀——因为活文档反复更新，日期前缀会过时误导。死文档（历史快照、审查报告，写完不再改）带日期前缀（`YYYYMMDD-中文名.扩展名`），方便按时间排序。每份文档开头第一行加状态头（草稿/有效/已归档）。

**定期清理**：每次发版做轻量清理——检查 `04-设计/` 是否有被替代的旧版（移到 `已归档/`）、检查 `README.md` 实现状态表是否与代码现状一致、检查 `CHANGELOG.md` 的 `[未发布]` 节点是否转为正式版本号。不积压到项目末期集中清理。

**不做的事**：不主动写设计文档、不主动更新功能范围清单、不主动写测试规范、不记录琐碎调整、不在对话中途打断用户。
