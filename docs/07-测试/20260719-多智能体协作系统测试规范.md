## 一、文档说明

### 1.1 测试目的

本规范用于系统性验证多智能体协作系统的所有功能模块，确保：

1. **功能完整性**：每个工具、每个调度流程、每个 UI 元素都按设计工作
2. **协作正确性**：主智能体 → 子智能体 → 主智能体的任务流转无遗漏
3. **可靠性**：异常场景（API 错误、非法参数、权限越界）有明确处理，不卡死
4. **防幻觉能力**：智能体无法编造文件路径、任务 ID、工具调用结果
5. **回归保护**：每次修改后，按本规范回归核心路径，避免"改一处坏一处"

### 1.2 测试分类

| 类别 | 说明 | 执行时机 |
|---|---|---|
| **冒烟测试** | 核心路径快速验证（第 4、5、14 章） | 每次修改后必做 |
| **功能测试** | 各模块独立功能验证（第 6-11 章） | 新功能上线时 |
| **异常测试** | 错误场景与边界条件（第 12 章） | 版本发布前 |
| **UI 测试** | 前端展示与交互（第 13 章） | 前端改动后 |
| **端到端测试** | 完整业务场景（第 14 章） | 版本发布前 |

### 1.3 测试通过标准

- 冒烟测试 100% 通过
- 功能测试通过率 ≥ 95%
- 异常测试 100% 通过（异常场景必须有明确处理，不能卡死或静默失败）
- 端到端测试 100% 通过

## 二、测试环境准备

### 2.1 环境检查清单

测试前必须确认以下环境就绪：

- [ ] **ENV-01** Node.js v24.15.0 已安装（`node -v` 验证）
- [ ] **ENV-02** Python 3.10.6 已安装（`python --version` 验证）
- [ ] **ENV-03** 后端依赖已安装（`pip install -r backend/requirements.txt`）
- [ ] **ENV-04** 前端依赖已安装（`cd frontend && npm install`）
- [ ] **ENV-05** 后端能正常启动（`cd backend && python main.py`，访问 `http://localhost:8000/api/health` 返回 `{"status":"ok"}`）
- [ ] **ENV-06** 前端能正常构建（`cd frontend && npm run build` 无报错）
- [ ] **ENV-07** 至少一个 API 配置可用（通义千问 / DeepSeek / Kimi 等，需有有效 api_key）
- [ ] **ENV-08** 浏览器能访问 `http://localhost:8000`，看到聊天界面

### 2.2 测试数据准备

- [ ] **DATA-01** 创建一个名为「测试项目」的新项目（用于功能测试）
- [ ] **DATA-02** 项目根目录设置为 `_test_template` 或新建空目录
- [ ] **DATA-03** 全局 API 配置已填写并测试通过（`/api/config/test` 返回成功）
- [ ] **DATA-04** （可选）创建一个模型配置 `cfg_test`，用于子智能体单独配模型

### 2.3 测试前清理

每次开始新一轮测试前，建议清理上一轮的残留数据：

- [ ] **CLR-01** 删除测试项目的所有子智能体（通过主智能体 `remove_team_member` 或管理弹窗）
- [ ] **CLR-02** 清空测试项目的消息（顶部「开启新任务」按钮）
- [ ] **CLR-03** 删除 `.agent/tasks/` 下所有任务文件夹
- [ ] **CLR-04** 删除 `deliverables/` 下所有产出文件
- [ ] **CLR-05** 重启后端（确保代码改动生效）
