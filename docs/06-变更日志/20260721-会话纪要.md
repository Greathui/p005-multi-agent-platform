| 新增 `_restorePartialMessages` | 从 localStorage 恢复之前保存的消息，恢复后删除 key，避免磁盘占用 |

## 轮次 18：修复后台运行方案（事件缓存+重放替换 swap 机制）

**做的事**：
- 用户反馈"点出去后再进来消息不能及时正常显示"
- 分析发现 `_swapToStream`/`_swapBackFromStream` 方案有严重问题：`handleSSEEvent` 中的 `done` 事件调用异步 `this.loadMessages()`，在 swap 恢复后执行，用错误的 `currentConversationId` 加载消息
- 用"事件缓存+重放"方案完全替换 swap 机制

**新方案核心**：
1. `sendMessage` 时拍消息快照（深拷贝）
2. SSE 循环中：用户在看→正常处理；用户切走→事件缓存到 `pendingEvents`
3. SSE 结束时：如果用户不在看，静默重放到快照（更新快照供下次切回使用）
4. `selectConversation` 切回来时：从快照恢复→重放 `pendingEvents`→如果有 `done` 事件则 `loadMessages` 刷新

**删除的旧代码**：
- `_swapToStream` / `_swapBackFromStream`（引用交换机制）
- `_savePartialMessages` / `_restorePartialMessages`（localStorage 兜底）
- `_savedStreamState` 状态字段

**新增**：
- `_replayPendingEvents(convId)`：静默重放缓存事件到快照（保存/恢复全部状态，跳过 done 事件中的 loadMessages）

**改的文件**：
- `frontend/src/stores/agent.js`：`sendMessage` 重写 + `regenerateMessage` 重写 + `selectConversation` 重写 + `_replayPendingEvents` 新增 + 旧方法删除

## 轮次 19：综合竞态条件修复 + 后端事件循环阻塞修复

**做的事**：
- 用户反馈"聊天内容显示还是不正常"，要求改完自己测试，并整体评估代码中其他用户操作导致的 bug
- 通过浏览器实测定位到两个根因：前端多个竞态条件 + 后端 async generate() 阻塞事件循环
- 逐一修复 6 个 bug，全部通过浏览器端到端测试验证

**根因分析**：

| 根因 | 描述 |
|---|---|
| 后端事件循环阻塞 | `generate()` 是 `async def`，但内部调用同步 LLM 函数（同步 OpenAI 客户端），阻塞了 FastAPI 事件循环，导致 SSE 运行时所有其他 API 请求挂起 |
| 前端竞态条件 | `selectConversation` 的 `await` 期间 SSE 事件到达，push 到被清空的 `this.messages`，随后被快照覆盖丢失；`loadMessages` 无 convId 参数时用过期的 `currentConversationId` |

**修复清单**：

| 编号 | 问题 | 修复方式 | 改动文件 |
|---|---|---|---|
| BUG-后端 | `async def generate()` 阻塞事件循环 | 改为 `def generate()`（同步 generator），Starlette 自动在线程池运行；所有 `await asyncio.sleep()` 改为 `time.sleep()` | main.py（2 处 generate + 23 处 sleep） |
| BUG-类型 | `Generator[str, None]` 类型标注错误（typing.Generator 需要 3 个参数） | 改为 `Generator[str, None, None]` | main.py（2 处） |
| BUG-前端-1 | selectConversation 的 await 期间 SSE 事件丢失 | 新增 `_isSwitching` 标志，切换期间所有 SSE 事件走缓存路径（pendingEvents），切换完成后恢复直通 | agent.js |
| BUG-前端-2 | loadMessages 用过期的 currentConversationId 加载错误对话 | `loadMessages(convId)` 接受可选 convId，完成后检查 `currentConversationId === id` 才更新 `this.messages` | agent.js |
| BUG-前端-3 | finally 块无条件重置 loading，影响其他对话 | 仅当 `currentConversationId === convId` 时才重置 loading 和 streamingAgents | agent.js（sendMessage + regenerateMessage） |
| BUG-前端-4 | clearConversationMessages 未停止 SSE | 清空前先调用 `stopGeneration(id)` | agent.js |

**测试结果**（浏览器端到端，模拟活跃 SSE）：
- 第一次切换 A→B：✓（0 条消息，isSwitching 正常复位）
- 第一次切回 B→A：✓（28 条消息从快照恢复）
- 第二次切换 A→B：✓（0 条消息）
- 第二次切回 B→A：✓（20 条消息从后端加载——之前失败的场景）
- 第三次切换：✓（20 条消息，DOM 匹配）
- 后端非阻塞验证：✓（SSE 运行时 API 调用 2.02 秒成功返回）

**改的文件**：
- `backend/main.py`：2 处 `async def generate()` → `def generate()` + 23 处 `await asyncio.sleep()` → `time.sleep()` + 2 处 `Generator[str, None]` → `Generator[str, None, None]` + `typing` 导入补充 `Generator`
- `frontend/src/stores/agent.js`：新增 `_isSwitching` 状态 + `loadMessages` 加 convId 参数 + `selectConversation` 重写 + `sendMessage`/`regenerateMessage` SSE 循环加 `_isSwitching` 检查 + `handleSSEEvent` done 用 `_activeStreamConvId` + finally 块条件重置 loading + `clearConversationMessages` 加 stopGeneration

**遇到的问题和解决方式**：
- 浏览器点击发送按钮反复 "Index out of bounds"：绕过点击，直接用 `browser_evaluate` 调用 `store.sendMessage()` 测试

## 轮次 20：多项目并行时消息并发写入丢失修复

**做的事**：
- 用户提问：团队多开任务逻辑是否有 bug，团队成员能否同时在两个项目中工作
- 确认架构设计：每个项目的团队成员是独立副本（镜像员工），配置/文件/消息/上下文四层隔离，多项目并行本身是安全的
- 但发现 `messages.json` 的并发写入有**更新丢失（lost update）**风险：`generate()` 中 `load_messages → append → save_messages` 不是原子事务，两个项目同时 SSE 时后保存的会覆盖先保存的
- 设计并实施了原子消息操作方案，新增 5 个原子函数替换所有非原子的 load-modify-save 模式

**架构确认**：
- 新建项目时从团队复制智能体配置，每个子智能体生成新 `agent_id`（`agent_` + uuid）
- 同一团队的"张三"在项目A是 `agent_abc123`，在项目B是 `agent_def456`，完全独立
- 四层隔离：配置隔离（项目级副本）、文件隔离（独立 root_path）、消息隔离（按 conversation_id 区分）、上下文隔离（build_isolated_context 只给任务描述）

**并发 bug 根因**：
```
时序  项目A                    项目B                    messages.json
T1   load_messages() → 100条                          100条
T2                           load_messages() → 100条  100条
T3   append 消息A → 101条                             101条
T4   save_messages() → 写101条                        101条（含A不含B）
T5                           append 消息B → 101条     101条
T6                           save_messages() → 写101条 101条（含B不含A，A丢失！）
```
`_json_write_lock`（RLock）只保护单次文件写入原子性，不保护 load-modify-save 整个事务。

**修复方案**：新增 5 个原子消息操作函数，用 `_json_write_lock` 包住整个 load-modify-save 事务：

| 函数 | 用途 | 替换的旧模式 |
|---|---|---|
| `append_message(msg)` | 原子追加单条消息 | `messages.append(); save_messages()` |
| `append_messages(msgs)` | 原子批量追加 | `messages.extend(); save_messages()` |
| `remove_messages_by_ids(ids)` | 原子删除指定ID消息 | `[m for m in messages if...]; save_messages()` |
| `clear_conversation_messages_atomic(conv_id)` | 原子清空项目消息 | `load-filter-save` |
| `delete_conversation_messages_atomic(conv_id)` | 原子删除项目消息+孤儿 | `load-filter-save` |

**改造的保存点**（共 11 处）：

| 接口 | 保存点 | 改动 |
|---|---|---|
| `chat_stream` generate() | 用户消息 | `messages.append + save` → `append_message` |
| `chat_stream` generate() | direct模式子智能体回复 | 同上 |
| `chat_stream` generate() | direct模式主智能体@回复 | 同上 |
| `chat_stream` generate() | 链式调度第一轮回复 | 同上 |
| `chat_stream` generate() | 链式调度并行批次回复 | 同上 |
| `chat_stream` generate() | 主智能体收尾总结 | 同上 |
| `chat_stream` generate() | extra_rounds子智能体回复 | 同上 |
| `chat_stream` generate() | extra_rounds主智能体回复 | 同上 |
| `regenerate` generate() | 删除旧回复 | `save(new_messages)` → `remove_messages_by_ids` |
| `regenerate` generate() | direct模式新回复 | `msgs.append + save` → `append_message` |
| `regenerate` generate() | 链式调度新回复 | 同上 |
| `chat()` 非流式 | 用户消息+回复 | `messages.append + save` → `append_messages`（批量） |
| `clear_conversation_messages` | 清空项目消息 | `load-filter-save` → `clear_conversation_messages_atomic` |
| `delete_conversation` | 删除项目消息 | `load-filter-save` → `delete_conversation_messages_atomic` |

**测试结果**：
- 2项目同时发送：✅ 5条消息全部保存，无丢失
- 3项目×3轮并发（9次同时请求）：✅ 21条消息全部保存，零丢失

**改的文件**：
- `backend/main.py`：新增 5 个原子消息函数 + 改造 14 处保存点 + `chat()` 非流式接口改为批量追加

**架构意义**：
这次修复让"后台运行"功能真正可靠——多个项目同时 SSE 流式运行时，每个项目的消息都会完整保存，不会因为并发写入而丢失。之前的 `_json_write_lock` 只保护了文件写入的原子性（防损坏），现在扩展到保护整个 load-modify-save 事务（防丢失）。
