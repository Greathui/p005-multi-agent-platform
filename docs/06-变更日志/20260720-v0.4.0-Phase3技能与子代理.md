# v0.4.0 — Phase 3 技能绑定、子代理调用与蓝图技能分配

## 一、背景

v0.3.0 上线了技能系统、蓝图系统和元团队入口后，审查发现三大扩展系统与智能体之间的集成存在三块缺失：

1. **缺失1：智能体没有 `enabled_skills` 字段** — 技能虽然定义了，但无法绑定到具体智能体，工具合并和提示词注入逻辑也未实现
2. **缺失2：`invoke_sub_agent` 工具未实现** — 规划文档中的"临时子代理"能力只有工具定义骨架，没有执行逻辑，`can_invoke_sub_agent` 字段也未加入数据模型
3. **缺失3：蓝图应用时不绑定技能** — `api_apply_blueprint` 创建智能体时忽略了蓝图中的 `member.skills` 和 `member.can_invoke_sub_agent` 字段

此外，用户反馈 UI 布局可优化：希望把"管理智能体"入口合并到左上角智能体栏目，把"设置"按钮移到左下角。

本次版本（v0.4.0）完成这三块功能实现 + UI 调整，并通过端到端测试验证。

## 二、实现内容

### 2.1 缺失1：智能体技能绑定

**后端改动**：

- `AgentCreate` 类新增 `enabled_skills: list[str] = []`
- `AgentUpdate` 类新增 `enabled_skills: list[str] | None = None`
- `update_conversation_agent` API 增加字段更新逻辑
- `call_llm_stream_with_tools` 工具过滤逻辑增强：
  ```python
  # 合并技能附带的工具
  enabled_skills = agent.get("enabled_skills", []) if isinstance(agent, dict) else []
  if enabled_skills:
      for skill_id in enabled_skills:
          skill = sb_get_skill(skill_id)
          if skill:
              for t in skill.get("tools", []):
                  allowed_tool_names.add(t)
  
  # 注入技能的 system_prompt_fragment
  if enabled_skills:
      skill_fragments = []
      for skill_id in enabled_skills:
          skill = sb_get_skill(skill_id)
          if skill and skill.get("system_prompt_fragment"):
              skill_fragments.append(f"### 技能：{skill['name']}\n{skill['system_prompt_fragment']}")
      if skill_fragments:
          msgs[0]["content"] += "\n\n## 已启用技能\n" + "\n\n".join(skill_fragments)
  ```

**前端改动**：

- `Sidebar.vue` 智能体管理弹窗新增技能多选下拉框（`el-select multiple collapse-tags`）
- 新增 `loadSkills()` 函数从后端加载技能列表
- 新增 `handleAgentSkillsChange()` 函数处理技能变更并调用 `updateConversationAgent` API
- 新增 `initExtensionMaps()` 函数初始化技能和子代理开关映射

### 2.2 缺失2：子代理调用插件

**后端改动**：

- `AgentCreate`/`AgentUpdate` 新增 `can_invoke_sub_agent` 字段
- `invoke_sub_agent` 工具定义加入 `TOOL_DEFINITIONS`：
  ```python
  {
      "type": "function",
      "function": {
          "name": "invoke_sub_agent",
          "description": "创建一个临时子代理来处理子任务...",
          "parameters": {
              "type": "object",
              "properties": {
                  "sub_agent_name": {"type": "string", "description": "子代理的名称"},
                  "task_description": {"type": "string", "description": "给子代理的任务描述"},
                  "system_prompt": {"type": "string", "description": "子代理的系统提示词（可选）"},
                  "inherit_permissions": {"type": "boolean", "description": "是否继承当前智能体的文件权限", "default": True}
              },
              "required": ["sub_agent_name", "task_description"]
          }
      }
  }
  ```
- `execute_tool` 函数签名扩展：新增 `agent: dict = None` 和 `default_config_id: str = ""` 参数
- `execute_tool` 中实现完整的 `invoke_sub_agent` 执行逻辑：
  - 权限检查：只有 `can_invoke_sub_agent=True` 的智能体才能调用
  - 创建临时子代理对象（不写入 conversations.json）
  - **非递归**：子代理的 `can_invoke_sub_agent` 强制为 `False`
  - **不继承技能**：`enabled_skills=[]`（避免权限放大）
  - 继承父智能体的 `allowed_paths` 和 `model_config`
  - 使用 `collect_stream_sync` 同步收集子代理回复（最大 8 轮工具循环）
  - 汇总工具调用情况并返回结构化结果
- 工具过滤逻辑：`if agent.get("can_invoke_sub_agent", False): allowed_tool_names.add("invoke_sub_agent")`

**前端改动**：

- 智能体管理弹窗新增子代理开关（`el-switch`）
- 新增 `handleAgentSubAgentChange()` 函数处理开关变更

### 2.3 缺失3：蓝图应用技能分配

**后端改动**：

- `api_apply_blueprint` 函数在创建智能体时读取蓝图字段：

**后端改动**：

- 工具定义新增 `skills` 和 `can_invoke_sub_agent` 参数

  🔧 子代理调用: 已开启（可使用 invoke_sub_agent 工具）
  ```

### 2.5 UI 布局调整

**侧边栏调整**：

- 底部 footer 从"管理智能体"改为"⚙ 设置"按钮

- `handleSelectAgent` 函数增加 `manage` 命令分支

**跨组件状态管理**：

  })
  ```
