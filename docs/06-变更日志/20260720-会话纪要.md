## 轮次 8：讨论融合后表决、元团队评审按钮、专家自我进化

**做的事**：
- 讨论三个相互关联的问题：融合后是否需要专家表决、蓝图应用后加入元团队评审按钮、专家能否修改自己提示词
- 结论一：融合后不需要专家表决。表决在融合前（评审打分即表决），融合后由用户拍板。专家在融合阶段后再投票容易僵局，且削弱主智能体融合权威
- 结论二：元团队评审按钮有价值，补上"设计-使用-反馈"闭环。产出评审报告（诊断+建议），不直接改蓝图，可触发新一轮设计任务
- 结论三：专家不能自己改自己的提示词。自己改自己缺乏外部校验、容易过度修正、会破坏模型多样性。改用"外部模型提建议 + 用户确认版本升级"流程
- 用户澄清：说的"修改提示词"是专家自己的 system_prompt 自我进化，不是改蓝图里给项目团队成员写的提示词。元团队专家应该是常驻角色而非一次性角色
- 确认专家常驻 + 两层成长机制：第一层经验自动积累（每次任务后自动记录），第二层提示词版本升级（用户主导）

**确认的 5 项追加决策（D10-D14）**：
- D10: 融合后不表决，用户拍板
- D11: 元团队评审按钮加，产出评审报告
- D12: 专家不改自己提示词，走外部校验流程
- D13: 专家全局常驻，不是每任务创建
- D14: 两层成长机制（经验自动积累 + 用户主导提示词升级）

## 轮次 27：修复删除项目后 UI 不实时更新

**做的事**：
- 应用户反馈，删除团队内项目后侧边栏和团队主页的 UI 没有实时更新。
- 根因：`deleteConversation` action 只更新了 `this.conversations`，没有刷新 `this.teams`。而团队主页的项目列表来自 `store.teams` 中的 `projects` 字段（后端 `GET /api/teams` 返回），删除项目后 teams 数据没刷新。
- 修复：在 `deleteConversation` 中调用 `this.loadTeams()` 刷新团队数据。`loadTeams` 返回后，`store.currentTeam` getter 自动更新，TeamHome 的 `team` computed 响应式更新，侧边栏的 team-count 和项目列表也同步更新。

**改的文件**：
- 修改 `frontend/src/stores/agent.js`（deleteConversation action 新增 `this.loadTeams()` 调用）

**测试结果**：
- 删除团队内项目 → UI 实时更新：项目数 1→0，团队 count badge 显示 0，显示"暂无项目，点击＋创建"空状态
