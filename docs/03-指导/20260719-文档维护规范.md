# P005 文档维护规范

> 📌 状态：有效 | 版本：v1.2 | 最后更新：2026-07-21 | 变更记录：见 [CHANGELOG.md](../CHANGELOG.md)

本规范定义 P005 项目文档的编排、命名、状态管理和变更追踪规则。所有参与项目开发的人员（包括 AI 助手）都应遵守。

---

## 一、文档目录结构

```
docs/
├── README.md                         # 总索引（含实现状态 + 当前版本号）
├── CHANGELOG.md                       # 顶层变更日志（所有变更的流水账）
├── 01-规划/                          # 项目初期的技术规划和功能清单
├── 02-范围/                          # 功能范围清单、需求边界
├── 03-指导/                          # 给 AI 助手/开发者的修改指南（含本规范）
├── 04-设计/                          # 架构设计、模块设计文档
│   └── 已归档/                       # 被新版本替代的旧设计文档
├── 05-审查/                          # 代码审查、项目检查报告（历史快照）
├── 06-变更日志/                      # ⭐ 会话纪要、重大变更的详细背景记录（每次有产出的对话后及时维护）
└── 07-测试/                          # 测试规范
```

### 分类说明

| 目录 | 用途 | 更新频率 |
|---|---|---|
| `01-规划/` | 项目初期的整体规划 | 低（规划定下来后很少改） |
| `02-范围/` | 功能范围、需求边界 | 中（需求变更时改） |
| `03-指导/` | 给开发者/AI 的指引 | 中（流程变化时改） |
| `04-设计/` | 架构设计、模块设计 | 高（每次重大功能都应有设计） |
| `05-审查/` | 审查报告 | 不更新（历史快照） |
| `06-变更日志/` | ⭐ 会话纪要、重大变更的详细记录 | **每次有产出的对话后及时维护** |
| `07-测试/` | 测试规范 | 有测试需求才写 |

## 二、文件命名规则

### 活文档 vs 死文档

| 类型 | 特征 | 命名方式 | 示例 |
|---|---|---|---|
| **活文档** | 长期维护，反映当前状态（如代码架构说明、功能范围清单） | 语义化命名，**不带日期前缀**（反复更新时日期会过时误导） | `代码架构说明书.md` |
| **死文档** | 历史快照、审查报告，写完不再改 | 带日期前缀，方便按时间排序 | `20260721-测试报告-v2.0.md` |
| **追加式日志** | 只增不改（CHANGELOG、会话纪要） | 带日期前缀 | `20260721-会话纪要.md` |

### 死文档命名格式

```
YYYYMMDD-中文名.扩展名
```

- **日期**：取自文件创建时间（死文档不更新，日期固定）
- **中文名**：简洁明了，保留可读性
- **扩展名**：`.md`（文本类）/ `.html`（带图表的报告）

### 示例

```
✅ 活文档：代码架构说明书.md          （语义化命名，不带日期）
✅ 活文档：软件功能范围清单.md
✅ 死文档：20260717-技术规划.md       （带日期前缀）
✅ 死文档：20260718-项目检查报告.html
✅ 死文档：20260719-智能体分级与通用模板设计.md

❌ 活文档带日期：20260721-代码架构说明书.md  （活文档不应带日期前缀，会过时误导）
❌ 死文档无日期：技术规划.md                  （死文档应带日期前缀）
❌ 2026-07-17-技术规划.md                     （日期分隔符太多）
❌ tech-plan.md                                （英文名，不便阅读）
```

### 归档文档命名

被新版本替代的旧文档，移到 `已归档/` 子目录，并在文件名加版本后缀：

```
04-设计/已归档/20260719-智能体分级与通用模板设计_v0.3.md
```

## 三、文档状态机制

每个文档在**开头第一行**加状态标签：

### 状态类型

| 状态 | 含义 | 标记 |
|---|---|---|
| 草稿 | 还在讨论，未落地 | `> 📝 状态：草稿` |
| 有效 | 当前生效中 | `> 📌 状态：有效` |
| 已归档 | 被新版替代或功能已删 | `> 🗜️ 状态：已归档` |
| 历史快照 | 审查类，仅参考 | `> 🗜️ 状态：历史快照` |

### 完整状态头模板

```markdown
> 📌 状态：有效 | 版本：v1.0 | 最后更新：2026-07-19 | 变更记录：见 [CHANGELOG.md](../CHANGELOG.md)
```

包含字段：
- **状态**：草稿/有效/已归档/历史快照
- **版本**：文档自身的版本号（v1.0、v1.1 等）
- **最后更新**：YYYY-MM-DD
- **变更记录**：指向 CHANGELOG.md

### 状态转换

```
草稿 → 有效 → 已归档
           ↓
       （审查类）历史快照
```

- **草稿 → 有效**：设计落地、代码实现后
- **有效 → 已归档**：被新版本替代，或对应功能已删除
- **有效 → 历史快照**：仅适用于 `05-审查/` 下的报告

## 四、版本号机制

项目使用[语义化版本](https://semver.org/lang/zh-CN/)：

```
v主.次.修
```

| 版本类型 | 触发条件 | 示例 |
|---|---|---|
| **主版本** | 不兼容的大重构 | v0 → v1 |
| **次版本** | 新增功能 | v0.3 → v0.4 |
| **修订号** | bug 修复、小调整 | v0.3.0 → v0.3.1 |

### 发版流程

1. 在 `CHANGELOG.md` 把 `[未发布]` 改成 `[vX.Y.Z] - YYYY-MM-DD`
2. 在顶部新增一个空的 `[未发布]` 节点
3. 更新 `README.md` 的"当前版本"字段
4. 更新涉及文档的状态头（版本号、最后更新日期）

## 五、典型场景操作流程

### 场景 1：新增功能

> 例：要加"多项目并行管理"

1. **写设计文档**：`docs/04-设计/20260801-多项目管理设计.md`（状态：草稿）
2. **登记变更**：在 `CHANGELOG.md` 的 `[未发布]` 下"新增"分类加一行
3. **更新索引**：在 `README.md` 的文档清单加一行
4. **实现代码**：开发完成后，设计文档状态改"有效"
5. **发版**：把 `[未发布]` 改成 `[v0.4.0] - 2026-08-01`

### 场景 2：修改需求

> 例：角色模板从 5 个改成 8 个

1. **直接改设计文档**：不新建文件，改 `04-设计/20260719-智能体分级与通用模板设计.md`
2. **更新状态头**：改"最后更新"日期，文档版本号 +0.1
3. **登记变更**：在 `CHANGELOG.md` 的 `[未发布]` 下"变更"分类加一行
4. **大改动时归档旧版**：如果改动超过 50%，旧版移到 `04-设计/已归档/`，文件名加 `_v0.3` 后缀

### 场景 3：删除功能

> 例：下线"coordinator 模板"

1. **标记废弃**：设计文档里 coordinator 部分加"⚠️ 已废弃"标注
2. **登记变更**：在 `CHANGELOG.md` 的 `[未发布]` 下"移除"分类加一行
3. **整篇废弃时归档**：如果整篇文档只讲 coordinator，状态改"已归档"，移到 `已归档/`

### 场景 4：临时 hotfix

> 例：紧急修 bug

1. **不改文档**：代码级修复不需要设计文档
2. **只记 CHANGELOG**：在 `CHANGELOG.md` 的 `[未发布]` 下"修复"分类加一行
3. **下次发版统一归档**

### 场景 5：新审查报告

> 例：做一次新的代码审查

1. **新建报告**：`docs/05-审查/20260815-代码审查报告.html`（状态：历史快照）
2. **旧报告不删**：保留为历史快照，方便对比
3. **更新索引**：在 `README.md` 的文档清单加一行

## 六、变更记录检查清单

每次需求变更后，按此清单走：

- [ ] 新增/修改了代码 → 检查是否需要更新设计文档
- [ ] 设计文档有改动 → 更新文档头部"最后更新"日期和版本号
- [ ] 任何变更 → 在 `CHANGELOG.md` 的 `[未发布]` 节点登记
- [ ] 新建文档 → 在 `README.md` 文档清单加一行
- [ ] 删除/废弃功能 → 旧文档状态改"已归档"，移到 `已归档/`
- [ ] 发版 → `CHANGELOG.md` 的 `[未发布]` 改成 `[vX.Y.Z] - 日期`，更新 `README.md` 当前版本

## 七、CHANGELOG.md 记录规范

### 分类

| 分类 | 用途 |
|---|---|
| `新增` | 新加的功能 |
| `变更` | 对已有功能的修改 |
| `废弃` | 即将移除的功能（下版本删） |
| `移除` | 本版本已删除的功能 |
| `修复` | bug 修复 |
| `文档` | 文档相关改动 |

### 格式

```markdown
## [未发布]

### 新增
- **功能名**：简短描述（必要时加详细链接）

### 变更
- **功能名**：旧 → 新，原因

### 修复
- **问题**：根因和修复方式
```

### 大改动单独建文档

如果某次变更影响范围大（涉及多个模块、架构调整），在 `docs/06-变更日志/` 建详细文档：

```
docs/06-变更日志/20260801-多项目管理改造.md
```

内容包含：背景、方案对比、决策理由、影响范围、迁移步骤。CHANGELOG 里加一行指向它。

## 八、与代码的对应关系

文档和代码的对应关系通过 `README.md` 的"实现状态表"维护：

| 日期 | 类型 | 文档 | 实现状态 | 链接 |
|---|---|---|---|---|
| 7/17 | 规划 | 技术规划（F1-F8） | ✅ 全部已实现 | ... |

- **实现状态**：✅ 已实现 / 📌 持续更新 / 🗜️ 历史快照 / ⏳ 待实施
- 代码改了但文档没更新 = 技术债，应在下次迭代补上

## 九、单一权威来源

每个知识点在文档体系中只能有一个权威文档，其他文档引用但不复制内容。

### 权威源对照表

维护在 `docs/README.md` 的"文档清单与实现状态"表中。新文档创建前，先搜索 `docs/` 目录是否已有同类文档。如果有，直接更新原有文档，不要新建。

### 典型权威源

| 主题 | 权威文档 | 其他文档的做法 |
|---|---|---|
| 项目整体约束与红线 | `AGENTS.md` | 引用，不重复列举 |
| 功能范围 | `docs/02-范围/` 下的最新版 | 旧版移到 `05-审查/` 归档 |
| API 接口定义 | `docs/04-设计/API接口清单.md` | 代码架构说明里只写模块划分，不展开接口 |
| 代码架构 | `docs/04-设计/代码架构说明书.md` | AGENTS.md 只写关键概念，不展开架构 |

### 避免重复

同一信息不要同时出现在 AGENTS.md、设计文档、代码说明三处且内容冲突。AGENTS.md 只写关键约束，细节指向对应权威文档。

## 十、定期清理

每次发版时做一次轻量清理，不要积压到项目末期集中处理。

### 清理清单

- 检查 `04-设计/` 是否有被替代的旧版（移到 `已归档/`）
- 检查 `README.md` 的实现状态表是否与代码现状一致
- 检查 `CHANGELOG.md` 的 `[未发布]` 节点是否需要转为正式版本号
- 检查是否有无日期前缀的死文档（应补上日期前缀或重命名为活文档）
- 检查是否有重复内容（同一信息出现在多份文档且内容冲突）

### 不要做的事

- 不要等到项目末期集中清理——积压的过时文档会让清理变成一项独立的大工程
- 不要只增不减——被替代的旧版应立即归档，不要留作"参考"造成混淆

---

## 十一、AI 助手协作约定

AI 助手在修改本项目时，除了遵守 `AGENTS.md`（项目根目录），还应：

1. **改代码前**：先读 `README.md` 找相关设计文档，了解现状
2. **改代码后**：按本规范的检查清单更新文档
3. **新增功能**：必须先建设计文档（状态：草稿），再写代码
4. **删除功能**：必须更新 CHANGELOG 和归档相关文档
5. **不能只改代码不更文档**：文档和代码一致性是硬性要求
