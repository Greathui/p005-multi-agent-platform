# P005 多智能体团队协作系统 — 软件功能范围清单

本文档基于对后端代码（`backend/main.py`、`meta_team_template.py`、`skill_blueprint.py`）、前端源码（`frontend/src/`）以及全部技术文档的完整梳理，汇总系统当前已实现的所有功能。文档面向项目维护者与潜在使用者，作为功能盘点、范围确认和后续规划的基线。

---

## 一、项目概述

P005 是一个网页版的「多智能体团队协作平台」，界面形态类似微信聊天框。用户通过一个主智能体（项目经理角色）给不同的 AI 智能体安排工作，每个智能体有独立的角色、工具和文件夹权限，可以并行或链式地协作完成复杂任务。

**目标用户**：非程序员个人用户，用于个人项目爱好（如小说创作、调研分析、代码审查等场景）。

**当前版本**：v0.4.0（2026-07-20 发布），技能绑定、子代理调用、蓝图技能分配、元团队常驻化、团队管理已上线，UI 布局已调整。

**技术栈**：
- 前端：Vue 3 `<script setup>` + Pinia + Element Plus + Vite + markdown-it + highlight.js
- 后端：Python 3.10 + FastAPI（单文件 `main.py` 约 7000 行）+ openai 库
- 存储：纯 JSON 文件（无数据库），运行时目录按项目隔离
- 部署：本地单机，后端监听 `0.0.0.0:8000`，前端构建产物由后端静态托管

**明确不做的边界**：手机端、用户登录、云端部署、多用户协作、复杂权限继承。这些不在当前规划范围内。

### 2.2 项目运行时目录结构

每个项目（conversation）绑定一个工作目录，首次设置时自动初始化以下四层结构：

```
项目根目录/
├── .agent/
│   ├── tasks/{task_id}/        # 任务隔离区（input/ 只读、output/ 可写、meta.json、TASK.md）
│   └── todos/{todo_id}.json    # TODO 清单
├── agent_work/{agent_id}/      # 智能体私有工作区（互不可见）
├── shared/                     # 共享文件区（所有智能体可读）
└── deliverables/               # 最终交付物（主智能体整理归档）
```

`.agent` 以点开头，Windows 默认不显示，但前端文件面板会显式展示，便于用户查看任务进度。

### 2.3 三级智能体配置架构

| 层级 | 存储位置 | 作用 | 修改方式 |
|---|---|---|---|
| 全局模板 | `data.json` | 仅 `main` 主智能体，作为所有新项目的模板 | 全局智能体管理 API |
| 项目级 | `conversations.json` | 每个项目独立的子智能体配置 | 主智能体通过 `create_team_member` 工具创建，或前端管理弹窗编辑 |
| 对话级 | 运行时 | 单次对话内的临时配置覆盖 | 前端会话级模型选择器、思考模式开关 |

新建项目时自动复制全局 `main` 模板作为初始配置，对话内修改仅影响当前项目，不污染全局模板。

### 3.3 智能体分级与角色模板

系统采用三层架构，参考 TRAE 的 L0/L1/L2 分级：

| 层级 | 身份 | 工具范围 | 权限 | 上下文 |
|---|---|---|---|---|
| L0 | 主智能体（项目经理） | 全部工具 | `["*"]` 全路径 | 完整历史 + 团队信息自动注入 |
| L1 | 子智能体（按模板） | 按模板限定 | 本职能 + shared + 任务目录 | 隔离上下文（仅任务描述 + input） |
| L2 | 执行者（组长派发） | 基础工具 | 本职能子集 + 自己 output | 单任务上下文 |

**五个通用能力模板**（与领域正交，领域通过 `domain_hint`/`style_guide` 注入）：

| 模板 | 定位 | 工具集 | 可写 |
|---|---|---|---|
| `creator` | 创作者，产出原创内容 | read_file, write_file, edit_file, grep, glob, submit_task_result | 是 |
| `reviewer` | 审阅者，只读 + 反馈 | read_file, grep, glob, submit_task_result | 否 |
| `analyst` | 分析者，产出分析报告 | read_file, write_file, edit_file, grep, glob, submit_task_result | 是 |
| `worker` | 通用执行者 | list_directory, read_file, write_file, create_directory, grep, glob, edit_file, submit_task_result | 是 |
| `coordinator` | 协调者（组长） | 子智能体工具 + `create_sub_task` | 是 |

模板定义「能做什么」，领域参数定义「做什么」。`create_team_member` 支持 `name`、`role`、`template`、`system_prompt`、`domain_hint`、`style_guide`、`avatar` 等参数。

### 3.4 工具系统

系统共定义 24 个工具（OpenAI function calling 格式），按职能分配：

**通用工具**（所有智能体可用）：

| 工具 | 功能 |
|---|---|
| `list_directory` | 列文件夹内容 |
| `read_file` | 读文件（>50000 字符截断） |
| `write_file` | 写文件（自动建父目录） |
| `create_directory` | 创建文件夹 |
| `grep` | 正则内容搜索（支持 glob 过滤、上下文行、上限 50 条） |
| `glob` | 文件名模式匹配（支持 `**/*`、`*.{py,js}`，上限 100） |
| `edit_file` | SEARCH/REPLACE 精确编辑（要求唯一匹配） |
| `submit_task_result` | 子智能体提交任务结果（含防幻觉校验） |

**主智能体专属工具**（共 13 个）：

| 工具 | 功能 |
|---|---|
| `init_project_structure` | 初始化项目四层目录 |
| `create_agent_workspace` | 为子智能体建私有工作区 |
| `create_task` | 创建任务并分配（自动建 input/output + 授权） |
| `grant_path_access` | 授予路径权限 |
| `revoke_path_access` | 撤销路径权限 |
| `move_file` | 移动文件（归档到 deliverables/，含回读校验） |
| `copy_file` | 复制文件（含回读校验） |
| `list_tasks` | 查询任务状态（all/pending/completed） |
| `create_todo_list` | 创建 TODO 清单（2-10 项，单对话最多 5 个） |
| `update_todo_status` | 更新 TODO 项状态 |
| `list_team_members` | 列团队成员 |
| `create_team_member` | 创建子智能体 |
| `update_team_member` | 更新子智能体 |
| `remove_team_member` | 移除子智能体 |

**协调者专属工具**：

| 工具 | 功能 |
|---|---|
| `create_sub_task` | L2 执行者派发（仅 coordinator 模板可用，防递归） |

**可选工具**：

| 工具 | 功能 |
|---|---|
| `invoke_sub_agent` | 调用临时子代理（需 `can_invoke_sub_agent=True`，非递归，最大 8 轮，不写入 conversations.json，继承父智能体权限和模型配置，不继承技能避免权限放大） |

**工具分配规则**（v0.4.0 完善）：
- 主智能体：可用全部 `TOOL_DEFINITIONS`
- 子智能体：按 `template` 字段对应工具集过滤（默认 worker），再合并 `enabled_skills` 技能工具，若 `can_invoke_sub_agent=True` 加入 `invoke_sub_agent`
- 协调者：子智能体工具 + `create_sub_task`（防递归：L2 用 worker 模板）
- 临时子代理：强制 `can_invoke_sub_agent=False`（非递归），`enabled_skills=[]`（不继承技能）
- 过滤逻辑在 `call_llm_stream_with_tools()`，执行层兜底拦截越权调用

**任务类型权限模板**（`create_task` 时按类型自动授权）：

| 类型 | 授予路径 | 输出目录 |
|---|---|---|
| `document` 文档编写 | docs/ | deliverables/docs/ |
| `code_review` 代码审查 | src/ 只读 | deliverables/reviews/ |
| `coding` 代码开发 | src/ | src/ |
| `data_analysis` 数据分析 | data/ | deliverables/analysis/ |
| `testing` 测试检查 | tests/、src/ 只读 | test_results/ |
| `general` 通用任务 | 无 | deliverables/ |

**通用核心规则**（CORE_RULES，8 条）自动注入所有智能体 system_prompt：立即行动原则、工具调用硬约束、真实性约束、工具使用策略、响应格式、角色边界、任务进度感知、跨轮续接。

### 3.5 大模型调用

**多服务商支持**（8 家预设）：

| 服务商 | 预设模型 |
|---|---|
| OpenAI | gpt-4o-mini、gpt-4o、gpt-4.1-mini、gpt-4.1 |
| 深度求索 DeepSeek | deepseek-chat、deepseek-reasoner |
| 通义千问 Qwen | qwen-turbo、plus、max、long |
| 月之暗面 Kimi | moonshot-v1-8k、32k、128k |
| 智谱清言 | glm-4-flash、glm-4、glm-4-plus、glm-4-air |
| 硅基流动 | 聚合 Qwen2.5、DeepSeek-V3、Llama-3.1 |
| 本地 Ollama | qwen2.5、llama3.1、deepseek-r1 |
| 自定义 | 兼容所有 OpenAI 格式接口 |

**配置优先级**（从高到低）：
1. 智能体自己的 `model_config.config_id`
2. 会话级 `default_config_id`（前端传入）
3. 全局默认配置 `load_api_config()`（兜底）
4. `thinking_override`（前端思考开关覆盖）

**流式响应能力**：
- `call_llm_stream()`：基础流式调用，逐 chunk 产出文本
- `call_llm_stream_with_tools()`：支持 function calling，最大 12 轮工具循环，产出 SSE 事件（token/tool_start/tool_end/error/done）
- `collect_stream_sync()`：同步收集流式响应（用于并行调度场景）
- 工具参数解析有四层容错：直接解析 → 移除 markdown 代码块 → 提取 `{...}` 块 → 空对象占位

**上下文窗口管理**：
- `context_window` 字段可配置（范围 4096~1048576，默认 131072）
- 截断阈值 = `context_window - 8000（输出预留）- 2000（安全边际）`，最小 30000
- 始终保留 system + 最近 3 轮（6 条）+ 当前消息
- Token 估算：中文 1.5 字/token，英文 4 字符/token
- 子智能体回复超 200 字时，给主智能体的历史只保留摘要（M1 机制）

**思考模式**：支持 `enable_thinking` 通过 `extra_body` 传入，部分服务商不支持时自动回退（去掉参数重试）。

### 3.6 任务调度

**直接模式**（`direct=True`）：
- 只调用指定智能体，不触发链式调度
- 子智能体 @主智能体 时自动触发主智能体响应（一轮，防循环）
- 主智能体回复保存时带 `reply_to` 字段标记回复对象
- 用于子线程私聊、重新生成单条回复

**链式调度**（`direct=False`）：
- BFS 队列实现，每元素 `{agent, task, from_agent}`
- `max_rounds=6` 防无限循环
- `processed_ids` 防同轮重复调用
- 第一轮单智能体：真正流式逐 token 输出
- 后续轮次：`ThreadPoolExecutor` 并发，`MAX_PARALLEL=5`
- 同一智能体多任务自动合并
- 每轮检查 @提及，刷新 conv_agents（支持刚创建的成员）
- 收尾逻辑：最后一轮非 main 时自动让 main 总结，扫描 task_xxx 幻觉引用，最多 8 轮追加调度，连续 2 轮无 @提及或无 pending 任务则结束

**并行调度**：
- `batch_start`/`batch_end` SSE 事件通知前端
- 前端显示「第 X 轮 · N 个智能体并行执行（m/n 完成）」指示器
- 同智能体多消息合并执行

**协调者两级调度**：
- `create_sub_task` 创建 L2 执行者（worker 模板，受限工具集）
- 自动创建任务文件夹 + 授权
- 继承组长 model_config
- L2 完成后通知组长，不直接找主智能体
- 防递归：L2 用 worker 模板（不含 create_sub_task），执行层校验 template=="coordinator"

**临时子代理**（`invoke_sub_agent`）：
- 创建临时子代理（不写入 conversations.json）
- 默认继承父智能体权限，可设为只读 shared/
- 非递归（子代理不能再调用子代理）
- 最大 8 轮工具循环
- 不继承技能（避免权限放大）

**上下文隔离**（`build_isolated_context`）：
- 子智能体不继承主智能体全部历史
- 只接收：自己的 system_prompt + 项目根目录 + 任务描述 + 原始用户需求 + 任务文件指引

### 3.7 任务管理

| 功能 | 说明 |
|---|---|
| 创建任务 | `create_task` 自动生成 task_id、创建 input/output 目录、写 meta.json 和 TASK.md、按类型授权 |
| 任务元数据 | id、agent_id、type、status（pending/completed）、created_at、completed_at、input_files、result_files |
| 提交结果 | `submit_task_result` 含三项校验：任务匹配、文件存在性、字节数验证，拒绝虚假提交 |
| 查询任务 | `list_tasks` 支持 all/pending/completed 过滤，按修改时间倒序，显示统计和 pending 提醒 |
| TODO 清单 | `create_todo_list`（2-10 项，单对话最多 5 个）、`update_todo_status`（pending/in_progress/completed） |
| TODO 强制提醒 | `check_pending_todos()` 主智能体每轮回前注入未完成项提醒 |
| 跨轮续接 | 收到「继续」等延续性指令时，主智能体第一反应查任务状态并续接未完成任务 |

### 3.8 权限控制

| 机制 | 说明 |
|---|---|
| 白名单路径 | `check_agent_path_access()` 检查 `allowed_paths`，main 默认全部权限，空或含 `*` 视为全权限（旧数据兼容） |
| 路径匹配 | 精确匹配或前缀匹配，处理 `\` 和 `/` 分隔符差异 |
| 任务目录隔离 | 每个任务独立 `.agent/tasks/{task_id}/{input,output}/`，input 只读、output 读写 |
| 写入保护 | 子智能体写入 input 目录会被拒绝 |
| 私有工作区 | `agent_work/{agent_id}/` 互不可访问 |
| 路径穿越防护 | `resolve_project_path()` 相对路径解析 + `relative_to(root)` 校验 + null 字符检测 + 类型校验 |
| HTTP 接口防护 | `_safe_resolve_path()` 越界抛 403 |
| 上传安全 | `sub_dir` 只允许简单目录名，拒绝 `..`、`\`、`/`；文件名只留 basename |
| 工具集过滤 | 主智能体全部工具，子智能体按 template + skills + can_invoke_sub_agent 过滤 |
| 执行层兜底 | 即使 LLM 误调管理工具，执行层也会拦截非主智能体的越权调用 |
| 元团队约束 | 元团队 main 禁止调用 create_team_member/create_todo_list/update_todo_status |

### 3.9 防幻觉机制

系统实现了四层防幻觉防御：

1. **提示词硬约束**：CORE_RULES 第 3 条真实性约束，禁止编造路径/任务 ID，禁止「假装」执行，禁止描述未调用工具的结果
2. **归档动作回读校验**：`move_file`/`copy_file` 执行后自动回读目标文件，返回存在性和字节数证据
3. **任务提交校验**：`submit_task_result` 严格校验 task_id 存在 + 归属 + pending 状态 + 结果文件真实性，全文件不存在则拒绝提交
4. **调度循环幻觉检测**：扫描主智能体回复中的 `task_xxx` 引用，校验 `.agent/tasks/` 下是否存在，不存在则发送 `hallucination_warning` SSE 事件并追加警告到回复

### 3.11 蓝图系统

蓝图是元团队的核心产出，连接元团队和项目团队的桥梁（JSON 格式）。

**数据模型**：id、name、description、icon、icon_bg、category、version、source（manual/meta/market/imported）、blueprint_data、apply_count。

**蓝图 Schema 顶层字段**：`blueprint_version`（固定 "1.0"）、`team_name`、`description`、`created_by`、`created_at`、`members[]`（2-8 个）、`tasks[]`（可选，含依赖关系）、`recommended_config`、`execution_tips`。

**协作模式**（members[].collaboration）：
- `star`：星型，向主智能体汇报（需 `report_to`）
- `direct`：直接协作，成员间 @对方（需 `with`）
- `parallel`：并行协作，多人独立工作

**两层校验**：
- 基础校验：顶层字段、members≥2、collaboration.type
- 深度校验：版本、成员 2-8 个、名称唯一、direct 模式 with 引用、任务 assignee 存在、依赖无环（DFS 环检测）

**蓝图 CRUD**：创建时同时保存为独立文件 `blueprints/{bp_id}.json` 便于导入导出。

**蓝图导入**：支持从 JSON 字典导入（source="imported"）或从文件路径导入（如 deliverables/team_blueprint.json）。

**蓝图应用**（`api_apply_blueprint`）：
1. 读取蓝图 blueprint_data 并校验
2. 创建新项目，复制全局 main 模板
3. 初始化目录结构
4. 按 members 创建子智能体（注入 system_prompt、协作提示、权限说明、skills、can_invoke_sub_agent）
5. 标记 `blueprint_source`
6. 增加蓝图 apply_count

**蓝图成员技能分配**（v0.4.0 新增）：
- 蓝图 `members[].skills` 字段指定成员绑定的技能 ID 列表
- 蓝图 `members[].can_invoke_sub_agent` 字段控制成员是否可调用子代理
- 应用蓝图时这两个字段会写入创建的智能体对象，实现蓝图级别的技能和子代理能力配置

**前端蓝图管理页**：统计条（本地数、总成员数、已应用次数）、新建/导入/搜索、蓝图卡片展示成员芯片/任务数/协作类型统计、应用/编辑/导出/删除操作、应用弹窗填项目名+工作目录。

### 3.12 元团队（v0.4.0 已重构为常驻功能模块）

> ⚠️ **v0.4.0 重大变更**：元团队已从"固定四成员项目类型"重构为"独立的常驻专家系统"。旧版"固定四步流程"已废弃，旧接口 `POST /api/conversations/create-meta-team` 返回 410 Gone。

**核心定位**：元团队不再是项目类型，而是独立的常驻功能模块，侧边栏有"团队设计"入口，与项目列表分离。专家常驻配置，通过经验积累实现自我进化。

**常驻专家**（默认 3 个，支持 CRUD）：

| 专家 | 头像 | 职责 | 经验积累 |
|---|---|---|---|
| 需求分析师 | 🔍 | 拆解需求、识别任务类型/规模/约束 | 每次设计任务后自动追加经验记录 |
| 团队架构师 | 🏗️ | 设计团队结构、协作模式、任务流程 | 每次设计任务后自动追加经验记录 |
| 提示词工程师 | ✍️ | 为每个成员编写 system_prompt | 每次设计任务后自动追加经验记录 |

**工作模式**：
- **deep 模式（默认）**：三阶段执行引擎
  1. **方案撰写**：3 个专家并行（ThreadPoolExecutor）各自产出方案
  2. **评审**：3 个专家并行评审其他人的方案
  3. **融合**：汇总所有方案和评审，产出最终 `team_blueprint.json`
  - 全程 SSE 流式推送进度，前端展示阶段控制
- **fast 模式**：单专家直接产出，不经过评审和融合

**产出物**：
- `proposals/{expert_id}.md`：各专家的方案
- `reviews/{expert_id}.md`：各专家的评审
- `fusion_decision.md`：融合决策记录
- `team_blueprint.json`：最终蓝图（可一键应用创建项目团队）

**评审闭环**（v0.4.0 新增）：
- 项目运行后，主聊天窗顶部有"元团队评审"按钮
- 采集 7 个维度的反馈（需求理解、团队结构、提示词质量、协作模式等）
- 3 个专家并行诊断，产出诊断对比报告
- 评审结果存入 `meta_team_reviews/`，作为专家经验的一部分

**专家管理**（侧边栏"团队设计"→专家卡片→详情）：
- 查看/编辑专家名称、模型配置、system_prompt
- 经验记录列表（时间线展示）
- 提示词版本历史与回退（回退前自动备份当前版本）
- 提示词优化（SSE 流式展示分析过程）
- 删除专家（至少保留 1 个）

**前端入口**：
- 侧边栏"团队设计"标签 → 团队设计首页（专家卡片网格 + 任务列表）
- 新建设计任务 → 任务详情页（SSE 对话 + 产出文件面板 + 一键全流程）
- 不再在"新建项目"弹窗中显示元团队选项

**实现状态**：Phase 1（常驻专家 + 基础对话）✅ / Phase 2（多专家并行 + 评审融合）✅ / Phase 3（评审闭环）✅ / Phase 4（诊断对比 + 版本回退）✅

### 3.15 上下文管理

| 机制 | 说明 |
|---|---|
| 按智能体隔离 | direct 模式下子智能体只看发给它的消息和自己的回复 |
| 子智能体回复压缩 | 给主智能体构建历史时，子智能体回复只保留前 200 字（M1 机制） |
| 条数截断 | 主智能体保留 system + 最近 3 轮（6 条）+ 当前消息 |
| Token 预算截断 | 总 token 超阈值时从最早开始删，阈值 = context_window - 8000 - 2000，最小 30000（M3 机制） |
| 上下文隔离 | 子智能体不继承主智能体全部历史，只接收任务描述 + input 指引（阶段 2） |
| 产出文件不入上下文 | 除非智能体主动调用 read_file 读取 |

## 五、数据存储结构

系统采用纯 JSON 文件存储，无数据库。所有数据文件位于 `backend/` 目录：

| 文件 | 用途 | 结构 |
|---|---|---|
| `data.json` | 全局智能体模板 | `{"agents": [main 配置]}` |
| `configs.json` | 多模型配置库 | `{"configs": [{id, name, provider, base_url, api_key, model, temperature, max_tokens, enable_thinking, context_window}]}` |
| `config.json` | 全局默认 API 配置（兜底） | `{"api": {base_url, api_key, model, temperature, max_tokens}}` |
| `conversations.json` | 项目列表 + 项目级智能体配置 | `{"conversations": [{id, title, root_path, created_at, updated_at, agents, structure_initialized, is_meta_team?, blueprint_source?}]}` |
| `messages.json` | 所有项目的聊天消息 | `{"messages": [{id, agent_id, role, content, time, conversation_id, reply_to?}]}` |
| `skills.json` | 技能库（内置 + 自定义） | `{"skills": [...]}` |
| `blueprints.json` | 蓝图库索引 | `{"blueprints": [...]}` |

**运行时生成的目录**（位于各项目根目录下）：
- `.agent/tasks/{task_id}/`：任务文件夹（meta.json、TASK.md、input/、output/）
- `.agent/todos/{todo_id}.json`：TODO 清单
- `agent_work/{agent_id}/`：智能体私有工作区
- `shared/`：共享文件区
- `deliverables/`：最终交付物
- `blueprints/{bp_id}.json`：蓝图独立文件（便于导入导出）

**前端 localStorage 持久化**：
- `p005_default_config_id`：会话级默认模型配置 ID
- `p005_enable_thinking`：会话级思考模式
- `model_history`：模型输入历史（最多 20 条）

## 六、安全机制汇总

| 机制类别 | 具体措施 |
|---|---|
| 路径安全 | 白名单校验、路径穿越防护（null 字符检测、relative_to 校验）、HTTP 接口越界抛 403 |
| 任务隔离 | 每任务独立目录、input 只读、output 读写、子智能体写入 input 被拒 |
| 工作区隔离 | `agent_work/{agent_id}/` 互不可访问 |
| 工具集过滤 | 主智能体全部工具，子智能体按 template 过滤，执行层兜底拦截越权调用 |
| 上传安全 | sub_dir 只允许简单目录名，文件名只留 basename |
| API 密钥保护 | 前端展示脱敏值（含 `***`），保存时检测脱敏值则不覆盖原 Key |
| 防递归 | L2 用 worker 模板（不含 create_sub_task），invoke_sub_agent 强制非递归 |
| 防幻觉 | 四层防御：提示词约束、归档回读校验、提交校验、调度循环检测 |
| CORS 限制 | 仅允许 `http://localhost:5173` |
| 文件大小限制 | read_file 超 50000 字符截断，上传单文件限 10MB |
| 搜索上限 | grep max_results=50，glob 上限 100 |

## 八、已知限制与边界

### 8.1 当前不支持的边界

- 手机端、用户登录、云端部署、多用户协作、复杂权限继承（明确不做）
- 技能市场和蓝图市场页面为占位，「安装」/「获取」按钮提示「功能即将上线」
- `create_meta_task` 工具在 skill_blueprint.py 声明为元团队技能工具，但 main.py 中尚未实现，调用会返回「未知工具」错误

### 8.2 已知风险点（多数已修复）

| 编号 | 风险 | 状态 |
|---|---|---|
| RISK-01 | 通义千问 function.arguments 非法 JSON | 已修复（四层容错解析） |
| RISK-02 | 脱敏值覆盖真实密钥 | 已修复（保存时检测脱敏值） |
| RISK-03 | conversations.json 格式损坏 | 已修复（迁移逻辑） |
| RISK-04 | 主智能体编造 task_id | 已修复（调度循环幻觉检测） |
| RISK-05 | 子智能体继承全部历史导致 token 爆炸 | 已修复（上下文隔离） |
| RISK-06 | `conv.get("agents") or load_agents()` 短路 | 已修复 |
| RISK-07 | enable_thinking 不被支持 | 已修复（自动回退） |
| RISK-08 | 子智能体越权访问项目外路径 | 高风险，需持续关注 |
| RISK-09 | 并发调度无上限 | 已修复（MAX_PARALLEL=5） |
| RISK-10 | 子智能体死循环互相 @ | 已缓解（max_rounds=6 + 收尾逻辑） |
