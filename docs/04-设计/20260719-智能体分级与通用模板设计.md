# 智能体分级与通用模板设计

> **文档目的**：为 P005 智能体协作平台设计一套领域无关的智能体分级与角色模板体系，解决复杂项目下智能体过多导致的上下文爆炸、角色边界模糊、调度效率低、角色质量不稳定等问题。
>
> **适用范围**：通用智能体协作平台（非代码专用工具）。支持小说创作、研究报告、教学设计、运营方案、软件开发等任意领域。
>
> **参考标杆**：TRAE 的三层智能体架构（L0 主智能体 / L1 专门子智能体 / L2 通用子智能体）。

---

## 一、背景与问题

### 1.1 现状

> ⚠️ 以下为 v0.1.0 时的现状描述，保留作为历史参考。v0.4.0 实际情况见下方"v0.4.0 更新说明"。

v0.1.0 时的两层平铺结构：

- **L0 主智能体**（项目经理）：当时拥有 19 个工具，完整对话上下文。
- **L1 子智能体**（平铺）：所有子智能体工具集相同，上下文继承自主智能体历史。

**v0.4.0 更新说明**：
- L0 主智能体现拥有 **24 个工具**（文件操作 7 + 项目管理 11 + 团队管理 4 + 提交 1 + 协调者 1 + 可选 1，详见 AGENTS.md 第七节）
- L1 子智能体按 `template` 字段（creator/reviewer/analyst/worker/coordinator）分配不同工具集
- 已实现上下文隔离（`build_isolated_context`）、并行调度（`MAX_PARALLEL=5`）、三级分组（coordinator）
- 新增技能系统、蓝图系统、子代理调用（`invoke_sub_agent`）、元团队常驻专家等扩展机制

### 1.2 复杂项目下的痛点

当项目需要的智能体数量超过 5 个（如全栈应用、长篇小说、大型研究报告）时，出现以下问题：

| 痛点 | 表现 | 影响 |
|---|---|---|
| **上下文爆炸** | 所有子智能体汇报都进入主智能体历史，做几轮就超 token | 响应变慢、成本升高、后期失忆 |
| **角色边界模糊** | 文档智能体和代码智能体工具集、权限完全相同 | 容易越界操作，产出混乱 |
| **调度效率低** | 串行链式调度（@mention → 转发 → 再转发） | 无法并行，耗时长 |
| **角色质量不稳定** | 主智能体临时编写的 system_prompt 质量参差 | 角色表现时好时坏 |
| **权限过宽** | 所有子智能体都能读写所有路径 | 安全隐患，误操作风险 |
| **重复造人** | 主智能体不查看现有成员就创建新角色 | 同一职责出现多个智能体 |

### 1.3 设计目标

- **领域无关**：模板不绑定具体职业（不预设"前端工程师""小说作者"），按能力特征分类
- **按需分级**：简单项目用单层即可，复杂项目可启用多级
- **工具策略内置**：模板预定义工具集和权限，不依赖主智能体临时发挥
- **上下文隔离**：子智能体只拿任务相关上下文，不继承全部历史
- **向后兼容**：现有 `create_team_member` 接口保留，新增 `template` 可选参数

## 二、设计原则

### 2.1 最小权限原则

每个智能体只授予完成任务所需的最小工具集和路径权限。

- 文档审阅员不需要 `write_file`（只读）
- 创作者不需要 `move_file`（归档由主智能体做）
- 执行者只能访问自己被分配的任务目录

### 2.2 能力分级原则

模板按**能力特征**分类，不按**具体职业**分类。领域信息通过参数注入。

- 模板定义"能做什么"（读写、只读、分析、执行）
- 领域定义"做什么"（写小说、写代码、写报告）

### 2.3 上下文隔离原则

子智能体调用时不继承主智能体的全部历史，只接收：
- 自己的 system_prompt
- 任务描述（TASK.md 内容）
- input 文件

完成后只返回 `submit_task_result` 的摘要，不把完整回复塞回主智能体历史。

### 2.4 角色模板化原则

预定义优质 prompt 骨架，主智能体创建成员时从模板选择并注入领域参数，不依赖临时发挥。

### 2.5 渐进式复杂度原则

- 简单项目（3-5 个智能体）：单层平铺即可
- 中等项目（6-10 个）：启用角色模板
- 复杂项目（10+ 个）：启用上下文隔离 + 并行调度
- 超大型（多团队）：启用三级分组（组长 → 组员）

## 三、智能体分级架构

### 3.1 三层架构总览

```
┌─────────────────────────────────────────────────┐
│  L0 主智能体（项目经理）                          │
│  - 全部工具（19个）                              │
│  - 完整对话上下文                                │
│  - 职责：规划、调度、归档、对用户负责              │
└─────────────┬───────────────────────────────────┘
              │ 派发任务
              ▼
┌─────────────────────────────────────────────────┐
│  L1 子智能体（按能力模板创建）                     │
│  - 受限工具集（按模板）                          │
│  - 隔离上下文（只拿任务描述）                      │
│  - 职责：执行具体任务，返回摘要                    │
│  - 类型：creator / reviewer / analyst / worker   │
└─────────────┬───────────────────────────────────┘
              │ （可选，复杂项目）
              ▼
┌─────────────────────────────────────────────────┐
│  L2 执行者（组长按需创建）                        │
│  - 基础工具（grep/glob/read/write/edit）         │
│  - 单任务上下文                                  │
│  - 职责：执行细分任务                            │
└─────────────────────────────────────────────────┘
```

### 3.2 各层职责

#### L0 主智能体（项目经理）

| 维度 | 说明 |
|---|---|
| **身份** | 项目经理，用户唯一对话入口 |
| **工具** | 全部 19 个（含管理工具、归档工具、搜索编辑工具） |
| **权限** | `["*"]` 全部路径 |
| **上下文** | 完整对话历史 + 自动注入的团队成员信息 |
| **职责** | 理解用户需求、规划任务、调度子智能体、归档产出、对用户负责 |
| **禁止** | 不亲自执行实质性任务（写文档、写代码），只做协调 |

#### L1 子智能体（按能力模板）

| 维度 | 说明 |
|---|---|
| **身份** | 按能力模板创建的专业执行者 |
| **工具** | 按模板限定（见第四章） |
| **权限** | 按模板限定（本职能目录 + shared + 任务目录） |
| **上下文** | 隔离上下文（只拿任务描述 + input） |
| **职责** | 执行主智能体分配的任务，产出文件到 output/，用 `submit_task_result` 提交 |
| **禁止** | 不能调用管理工具，不能访问其他智能体工作区 |

#### L2 执行者（组长派发，可选）

| 维度 | 说明 |
|---|---|
| **身份** | 组长按需创建的细分执行者 |
| **工具** | 基础工具（grep/glob/read/write/edit + submit_task_result） |
| **权限** | 本职能目录子集 + 自己的 output |
| **上下文** | 单任务上下文 |
| **职责** | 执行组长分配的细分任务 |
| **启用条件** | 仅复杂项目（10+ 智能体）启用 |

### 3.3 与 TRAE 架构的对比

| 维度 | TRAE | P005（本设计） |
|---|---|---|
| 分级依据 | 按能力（Explore/Plan/General） | 按能力 + 按职能（双维度） |
| 子智能体类型 | 预定义 3 种 | 预定义 5 种能力模板 |
| 上下文隔离 | 强（独立上下文） | 引入（L1 只拿任务描述） |
| 并行调度 | 支持 | 分阶段引入 |
| 工具策略 | 类型内置 | 模板内置 |
| 领域适配 | 代码专用 | 通用（领域参数注入） |

## 四、通用能力模板设计

### 4.1 模板设计哲学

**核心理念**：模板定义"能做什么"，领域定义"做什么"。

- 模板 = 能力特征（工具集、权限、协作方式、prompt 骨架）
- 领域 = 注入参数（role_description、domain_hint、style_guide）

**举例**：
- 小说作者 = `creator` 模板 + `domain_hint="小说章节"` + `style_guide="金庸武侠风格"`
- 前端工程师 = `creator` 模板 + `domain_hint="React 组件"` + `style_guide="遵循 Airbnb 规范"`
- 试读读者 = `reviewer` 模板 + `domain_hint="小说"` + `style_guide="从读者视角"`
- Code Reviewer = `reviewer` 模板 + `domain_hint="代码"` + `style_guide="严格但建设性"`

### 4.2 五个通用能力模板

#### 模板 1：creator（创作者）

| 维度 | 内容 |
|---|---|
| **定位** | 产出原创内容（文本、代码、文档、故事等） |
| **工具** | `read_file, write_file, edit_file, grep, glob, submit_task_result` |
| **权限** | 读写 `agent_work/{id}/`, `shared/`, `.agent/tasks/{task_id}/output/` |
| **禁止** | 删除文件、访问其他智能体工作区、调用管理工具 |
| **适用场景** | 小说作者、文案撰写、报告撰稿、前端开发、后端开发、教学设计 |

**system_prompt 骨架**：

```
你是「{name}」，{role_description}。你是项目团队的一员，由主智能体调度。

## 你的能力
- 创作原创内容（{domain_hint}）
- 修改完善已有内容
- 按要求格式产出

## 工具使用建议
- 找参考资料 → 用 grep（按内容）或 glob（按文件名）
- 改大文件的几行 → 用 edit_file（省 token 且安全）
- 新建文件 / 大改重写 → 用 write_file
- 不要用 list_directory 逐层遍历找文件，用 glob 代替

## 工作流程
1. 读取任务 TASK.md 了解需求
2. 读取 input/ 下的参考资料（只读，不要写入）
3. 创作产出，写到 output/ 目录
4. 用 submit_task_result 提交结果

## 原则
- 内容原创，不抄袭
- 遵循 {style_guide}
- 不确定的细节主动说明，不编造
- 如需反馈或澄清，在回复里写「@主智能体」
```

#### 模板 2：reviewer（审阅者）

| 维度 | 内容 |
|---|---|
| **定位** | 只读 + 提反馈意见（不修改原作） |
| **工具** | `read_file, grep, glob, submit_task_result` |
| **权限** | 只读（无写权限） |
| **禁止** | 任何写操作、删除操作 |
| **适用场景** | 试读读者、编辑审稿、Code Review、质量检验、用户视角测试 |

**system_prompt 骨架**：

```
你是「{name}」，{role_description}。你是项目团队的审阅者，由主智能体调度。

## 你的能力
- 阅读内容并给出专业反馈
- 从{style_guide}视角发现问题
- 不修改原作，只提建议

## 工具使用建议
- 找特定内容 → 用 grep（按内容）或 glob（按文件名）
- 读文件 → 用 read_file，指定具体路径
- 你没有写文件的权限，反馈通过 submit_task_result 提交

## 工作流程
1. 读取任务 TASK.md 了解审阅要求
2. 读取 input/ 下待审阅的内容
3. 从{style_guide}角度给出审阅意见：
   - 具体问题（引用原文位置）
   - 改进建议（可操作，不空泛）
   - 亮点（值得肯定的地方）
4. 用 submit_task_result 提交审阅报告

## 原则
- 只读，不修改原作
- 反馈具体，引用原文位置
- 建议可操作，不空泛
- 客观公正，先肯定亮点再提问题
```

#### 模板 3：analyst（分析者）

| 维度 | 内容 |
|---|---|
| **定位** | 分析数据/资料，产出分析报告 |
| **工具** | `read_file, grep, glob, write_file, edit_file, submit_task_result` |
| **权限** | 只读输入数据 + 写分析报告到 output/ |
| **禁止** | 修改原始数据 |
| **适用场景** | 数据分析、市场调研、竞品分析、可行性研究、学术研究 |

**system_prompt 骨架**：

```
你是「{name}」，{role_description}。你是项目团队的分析者，由主智能体调度。

## 你的能力
- 分析{domain_hint}并给出结论
- 产出结构化的分析报告
- 基于证据，不臆测

## 工具使用建议
- 找数据/资料 → 用 grep（按内容）或 glob（按文件名）
- 分析报告写到 output/ 目录
- 改报告用 edit_file，新建用 write_file

## 工作流程
1. 读取任务 TASK.md 了解分析目标
2. 读取 input/ 下的原始数据/资料（只读）
3. 分析并产出报告：
   - 明确分析方法和数据来源
   - 给出具体结论（有数据支撑）
   - 标注不确定性
4. 写到 output/，用 submit_task_result 提交

## 原则
- 基于证据，不臆测
- 结论明确，有数据支撑
- 标注数据来源和分析方法
- 不确定的地方明确说明
```

#### 模板 4：worker（通用执行者）

| 维度 | 内容 |
|---|---|
| **定位** | 通用执行，不限定具体能力 |
| **工具** | 全部子智能体工具（8 个） |
| **权限** | 读写限定路径（由主智能体授予） |
| **禁止** | 管理工具、归档工具 |
| **适用场景** | 不属于上述三类的任意执行任务（如：格式转换、文件整理、翻译、校对） |

**system_prompt 骨架**：

```
你是「{name}」，{role_description}。你是项目团队的执行者，由主智能体调度。

## 你的能力
- 执行{domain_hint}相关任务
- 灵活应对各类执行需求

## 工具使用建议
- 找文件 → 用 grep 或 glob
- 改文件 → 用 edit_file（小改）或 write_file（重写）
- 新建文件 → 用 write_file

## 工作流程
1. 读取任务 TASK.md 了解任务详情
2. 读取 input/ 下的输入文件
3. 执行任务，产出写到 output/
4. 用 submit_task_result 提交结果

## 原则
- 严格按任务要求执行
- 遵循 {style_guide}
- 不确定的地方主动询问（@主智能体）
- 完成后清晰说明做了什么、结果在哪里
```

#### 模板 5：coordinator（协调者，可选）

| 维度 | 内容 |
|---|---|
| **定位** | 组长，能派发任务给组员（仅复杂项目启用） |
| **工具** | 子智能体工具 + `create_sub_task`（派发给组员） |
| **权限** | 跨目录（本职能全部子目录） |
| **禁止** | 创建团队成员（只能创建任务） |
| **适用场景** | 大型项目分组（如：开发组长管前端+后端+测试） |
| **启用条件** | 仅复杂项目（10+ 智能体） |

**system_prompt 骨架**：

```
你是「{name}」，{role_description}。你是项目团队的{domain_hint}组长，由主智能体调度。

## 你的能力
- 管理本组（{domain_hint}）的任务执行
- 将大任务拆分成子任务派发给组员
- 汇总组员产出，向主智能体汇报

## 工作流程
1. 接收主智能体分配的大任务
2. 拆分成子任务，用 create_sub_task 派发给组员
3. 跟踪进度，汇总产出
4. 用 submit_task_result 向主智能体汇报整体结果

## 原则
- 只管理本组，不越界
- 拆分任务要明确，每个子任务有清晰交付物
- 汇总要简洁，突出关键结果
```

### 4.3 模板与领域的正交关系

模板与领域完全正交，任意模板可适配任意领域：

| 领域 \ 模板 | creator | reviewer | analyst | worker |
|---|---|---|---|---|
| **小说创作** | 作者 | 读者/编辑 | — | 校对员 |
| **研究报告** | 撰稿人 | 审稿人 | 分析师 | 资料整理 |
| **软件开发** | 开发工程师 | Code Reviewer | — | 测试工程师 |
| **教学设计** | 教案撰写 | 教学评审 | 学情分析 | 课件制作 |
| **运营方案** | 方案撰写 | 方案评审 | 数据分析 | 执行跟进 |
| **翻译** | 译者 | 校对 | — | 术语整理 |

### 4.4 模板选择决策树

```
用户需求
   │
   ├─ 产出新内容（文本/代码/文档）→ creator
   │
   ├─ 审阅已有内容并提反馈 → reviewer
   │
   ├─ 分析数据/资料给结论 → analyst
   │
   ├─ 通用执行任务（不属于上述）→ worker
   │
   └─ 管理一组人协作（复杂项目）→ coordinator
```

#### `create_team_member` 调用参数（扩展后）

```python
create_team_member(
    name: str,              # 必填，成员名字
    role: str,              # 必填，角色描述（如"武侠小说作者"）
    template: str = None,   # 可选，模板名（creator/reviewer/analyst/worker/coordinator）
    system_prompt: str = None,  # 可选，完全自定义（与 template 二选一）
    domain_hint: str = None,    # 可选，领域提示（如"小说章节"）
    style_guide: str = None,    # 可选，风格指南（如"金庸武侠风格"）
    avatar: str = "🤖",     # 可选，头像
)
```

**参数优先级**：
1. 如果传 `template`：使用模板 + 注入 `domain_hint`/`style_guide`，自动生成 system_prompt
2. 如果传 `system_prompt`：完全自定义（高级用户）
3. 两者都不传：报错（必须二选一）

### 5.2 工具 schema 过滤

当前 `call_llm_stream_with_tools` 已根据 `agent_id` 过滤工具 schema（主智能体看全部，子智能体看 `SUB_AGENT_TOOLS`）。

**扩展**：子智能体的工具 schema 进一步按模板的 `tools` 字段过滤：

```python
if agent_id == "main":
    tools = TOOL_DEFINITIONS
else:
    # 获取该智能体的模板
    template_name = agent.get("template", "worker")  # 默认 worker
    template = ROLE_TEMPLATES.get(template_name, ROLE_TEMPLATES["worker"])
    allowed_tools = template["tools"]
    tools = [t for t in TOOL_DEFINITIONS if t["function"]["name"] in allowed_tools]
```

### 5.3 权限路径生成

根据模板的 `allowed_paths_type` 自动生成权限路径：

```python
def generate_allowed_paths(template_type, agent_id, root, task_id=None):
    paths = []
    # 所有子智能体都有自己的工作区
    work_dir = root / "agent_work" / agent_id
    paths.append(str(work_dir.resolve()))
    # 共享区
    shared_dir = root / "shared"
    paths.append(str(shared_dir.resolve()))
    # 任务目录
    tasks_dir = root / ".agent" / "tasks"
    paths.append(str(tasks_dir.resolve()))
    
    if template_type == "readonly":
        # reviewer：只读，不加写权限（通过 check_path_permission 控制）
        pass
    elif template_type == "output":
        # creator/worker：可写 output
        pass  # 已包含 tasks_dir
    elif template_type == "analysis":
        # analyst：只读 input，写 output
        pass
    elif template_type == "group":
        # coordinator：本组目录
        group_dir = root / "deliverables" / agent_id
        paths.append(str(group_dir.resolve()))
    
    return paths
```

### 5.4 上下文隔离（阶段 2 实现）

子智能体调用时不继承主智能体历史，改为：

```python
# 旧：继承主智能体历史
sub_history = build_conversation_history(conv_id, sub_prompt, task, max_history=20, ...)

# 新：隔离上下文，只传任务描述
sub_history = [
    {"role": "system", "content": sub_prompt},
    {"role": "user", "content": f"【任务】{task_description}\n\n请查看 .agent/tasks/{task_id}/TASK.md 了解详情。"}
]
```

完成后只返回 `submit_task_result` 的摘要，不把完整回复塞回主智能体历史。

### 5.5 并行调度（阶段 3 实现）

同层多个子智能体可并行执行：

```python
# 旧：串行链式
while queue:
    item = queue.pop(0)
    result = call_agent(item)

# 新：并行执行（同批多个）
current_batch = [queue.pop(0) for _ in range(min(len(queue), max_parallel))]
with ThreadPoolExecutor(max_workers=len(current_batch)) as executor:
    futures = [executor.submit(call_agent, item) for item in current_batch]
    results = [f.result() for f in futures]
```

## 十一、阶段 2/3/4 实现说明（v1.1 增补）

> 以下为 2026-07-19 完成的阶段 2/3/4 实现细节，供后续维护参考。

### 11.1 阶段 2：上下文隔离（解决 token 爆炸）

**核心改动**：新增 `build_isolated_context()` 函数，子智能体不再继承主智能体全部历史。

| 对比项 | 改造前 | 改造后 |
|---|---|---|
| 子智能体上下文来源 | `build_conversation_history()` 加载全部对话历史 | `build_isolated_context()` 只生成 system + user 两条 |
| 子智能体可见内容 | 主智能体历史 + 其他子智能体回复 + 任务描述 | 仅自己的 system_prompt + 任务描述 + 原始用户需求 |
| 跨智能体信息传递 | 通过历史消息注入（token 爆炸） | 通过文件传递（output→input） |
| 主智能体上下文 | 完整历史（保留） | 完整历史（保留，需全局视角） |

**关键函数**：`build_isolated_context(system_prompt, task_description, root_path, from_agent_name, original_user_msg, task_id)`

**结果回注优化**：主智能体收尾总结时，对超过 2000 字的子智能体回复做截断，避免超长内容塞入主智能体历史。

**主智能体提示词增强**：在 `get_team_context_for_main()` 中补充上下文隔离说明，提醒主智能体派发任务时写清楚背景信息（因子智能体看不到完整对话）。

### 11.2 阶段 3：并行调度（效率提升）

**核心改动**：增加 `MAX_PARALLEL` 配置 + 同智能体多消息合并 + 批次 SSE 事件。

| 能力 | 说明 |
|---|---|
| `MAX_PARALLEL = 5` | 限制同轮最大并发数，避免 API 限流和资源耗尽 |
| 同智能体多消息合并 | 同一轮中同一智能体被多个来源 @mention 时，合并任务描述为一条，避免只处理第一个 |
| `batch_start` 事件 | 通知前端本批并行执行的智能体列表（round/count/parallel/agents） |
| `batch_end` 事件 | 通知前端本批执行完成 |

**前端可视化**：ChatWindow 新增并行批次状态条，显示"第 N 轮 · M 个智能体并行执行（X/M 完成）"及参与智能体名称芯片。`activeBatch` 状态在 store 中管理，`agent_end` 时累加完成计数。

**合并逻辑的价值**：不仅提升并行效率，更是阶段 4 的基础——当多个 L2 执行者同时 @coordinator 汇报时，coordinator 能一次性收到所有结果而非只看第一个。

### 11.3 阶段 4：三级分组（超大型项目）

**核心改动**：新增 `create_sub_task` 工具，coordinator 可创建 L2 执行者并派发子任务。

**三级架构**：
- **L0 主智能体**（项目经理）：完整上下文，全部工具，负责规划/归档
- **L1 协调者**（组长，coordinator 模板）：隔离上下文，含 `create_sub_task`，管理本组
- **L2 执行者**（worker 模板）：隔离上下文，受限工具集，向 L1 汇报

**`create_sub_task` 工具**：
- 仅 coordinator 模板可用（execute_tool 内校验 template）
- 自动创建 L2 执行者（worker 模板 + level=2 + parent_agent_id）
- 自动创建任务目录（.agent/tasks/{task_id}/，含 input/output/meta.json/TASK.md）
- 自动授权（本任务目录 + 私有工作区 + shared + 任务类型相关路径）
- L2 执行者写入项目级 conversations.json，可被 @mention 调度

**智能路由**：`submit_task_result` 根据 `parent_agent_id` 自动判断汇报对象——L2 完成后 @组长，L1 完成后 @主智能体。

**防递归保护**：L2 执行者使用 worker 模板（工具集不含 `create_sub_task`），且 execute_tool 内校验 template=="coordinator"，双重保险避免无限递归。

**调度循环适配**：现有 BFS 队列天然支持三级递归（main→coordinator→L2→coordinator→main），配合阶段 3 的多消息合并，coordinator 能汇总多个 L2 结果后统一向主智能体汇报。

**前端层级展示**：团队管理弹窗显示 L0/L1/L2 徽章（蓝/绿/紫），L2 执行者缩进显示并带紫色左边框；聊天窗口智能体芯片对 L2 显示紫色 L2 标签。
