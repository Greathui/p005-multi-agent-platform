# 元团队常驻化与工作模式重构方案

## 一、背景与问题

### 1.1 现状

元团队（Meta Team）当前作为"特殊类型的项目"存在，通过新建项目时选择"元团队规划"类型创建。固定四成员（需求分析师、团队架构师、提示词工程师、质量评审员），按固定四步流水线（需求分析 → 团队架构 → 提示词工程 → 质量评审）产出 `team_blueprint.json` 蓝图文件。

### 1.2 暴露的问题

经过实际使用，当前设计存在以下结构性问题：

**问题一：归类错误。** 元团队是"设计工具"而非"执行单元"，却与普通项目并列在左侧栏。普通项目有工作目录、文件产出、长生命周期；元团队只聊需求产出蓝图，生命周期短。两者性质不同，强行并列导致心智模型混乱。

**问题二：一次性浪费。** 元团队项目产出蓝图后即废弃，占用 conversations.json 一条记录。用户多建几个元团队，列表里堆积废项目。历史设计过程无法回溯，无法基于旧设计迭代。

**问题三：入口错位。** "新建项目"时选择"元团队规划"类型这个入口本身就别扭——用户想"设计团队"时，不应该走"新建项目"流程。

**问题四：流程刚性。** 固定四步流水线对简单需求过度设计（建个三人小说团队也要走完四步），对复杂需求又不够灵活（无法跳步、无法返工）。详见工作模式讨论（第三节）。

**问题五：蓝图与设计过程脱节。** 蓝图存于 `blueprints.json`，与产生它的元团队项目无关联。用户拿到蓝图后，无法回溯"这个团队结构当初是怎么设计出来的"。

### 1.3 重构目标

- 将元团队从"项目类型"重构为"常驻功能模块"
- 重新设计元团队的工作模式，提升灵活性和产出质量
- 保持与现有蓝图系统的兼容，不破坏已有蓝图数据

## 二、常驻化架构方案

### 2.1 左侧栏结构调整

```
左侧栏
├── 我的项目
│   ├── 小说创作项目
│   ├── 代码审查项目
│   └── ...
├── 团队设计              ← 元团队作为独立模块常驻
│   ├── 进行中
│   │   └── 科幻小说团队设计
│   ├── 已完成
│   │   ├── 小说创作团队设计 (已产出蓝图)
│   │   └── 调研团队设计 (已应用)
│   └── + 新建设计任务
└── 设置
    ├── 模型配置
    ├── 技能管理
    ├── 蓝图管理           ← 蓝图管理仍在设置页（管成果）
    └── ...
```

**设计要点：**
- "团队设计"模块下按状态分组显示设计任务，不混入普通项目列表
- 蓝图管理（查看/导入/导出/应用）仍在设置页，"团队设计"管过程，"蓝图管理"管成果
- 新建项目弹窗去掉"元团队规划"类型选项，入口收敛到"团队设计"模块

### 2.2 数据存储调整

backend/

**变更：conversations.json**
- 移除 `is_meta_team` 字段和相关逻辑
- 已有的元团队项目需迁移到 `meta_team_tasks/`（数据迁移脚本）

**不变：blueprints.json / blueprints/**
- 蓝图库结构保持不变
- 元团队产出的蓝图仍然进这个库，只是来源标记从 `meta` 改为 `meta_team_task`

### 2.3 元团队专家常驻与自我进化（已确认采用）

**决策：元团队专家全局常驻，不是每任务创建。**

专家作为全局角色定义，所有设计任务共用同一组专家。每次设计任务只切换上下文（当前需求 + 历史经验），不重新创建角色。专家有记忆、能成长，但提示词修改由用户主导。

**专家存储：**

新增 `backend/meta_team_config.json`，与普通项目智能体彻底分开：

```json
{
  "experts": [
    {
      "id": "expert_1",
      "name": "团队设计专家A",
      "system_prompt": "...（当前版本提示词）",
      "prompt_version": 2,
      "model_config": {"config_id": "cfg_xxx"},
      "created_at": "2026-07-20T...",
      "updated_at": "2026-07-20T...",
      "experience_log": [
        {
          "task_id": "task_001",
          "task_title": "科幻小说团队设计",
          "score_received": 7.5,
          "score_breakdown": {"团队结构": 8, "提示词质量": 7, "协作机制": 8, "需求匹配": 7, "成本效率": 7.5},
          "feedback_summary": "团队结构合理但提示词偏长",
          "fusion_adopted": ["成员配置方案", "协作模式设计"],
          "fusion_not_adopted": ["提示词风格过于冗长"],
          "created_at": "2026-07-20T..."
        }
      ]
    },
    {
      "id": "expert_2",
      "name": "团队设计专家B",
      "system_prompt": "...",
      "prompt_version": 1,
      "model_config": {"config_id": "cfg_yyy"},
      "experience_log": []
    }
  ],
  "default_expert_count": 3,
  "default_mode": "deep"
}
```

**两层成长机制：**

**第一层：经验自动积累（每次设计任务后自动执行，无需用户干预）**

设计任务完成后，系统自动为每个参与专家追加一条经验记录，包含：
- 本次方案收到的平均得分和其他专家的评分明细
- 其他专家的文字评审意见摘要
- 主智能体融合决策说明中，该专家方案的哪些部分被采纳、哪些没被采纳

专家在下一次设计任务时，能看到自己的全部历史经验记录作为参考。这一层不改专家的 system_prompt，只给专家加"记忆"。

**第二层：提示词版本升级（用户主导，非专家自动修改）**

当用户观察到某个专家"得分持续走低"或"方案质量不稳定"时，可以手动触发"提示词优化"流程：

1. 系统将该专家的经验记录 + 当前 system_prompt + 历史方案反馈给一个模型（主智能体或独立模型）
2. 该模型分析"这个专家的提示词哪里可以改进"，产出优化建议（附理由）
3. 用户审阅优化建议，确认后才升级到新版本（prompt_version + 1）
4. 新版本提示词用于后续设计任务

**为什么不让专家自己改自己的提示词：**
- 自己改自己缺乏外部校验，容易越改越偏（过度修正）
- 提示词是高度耦合的系统，改一处影响全局，专家难以判断全局影响
- 3 个专家都往"被融合采纳次数最多"的方向靠拢，会趋同化，破坏模型多样性这一核心优势
- 改提示词等于改蓝图设计能力，必须走有外部校验的流程

**专家常驻的其他影响：**
- 专家之间仍然完全独立，不交流，经验记录私有，各自独立成长，多样性得以保留
- 专家卡片上显示"平均得分""最近 N 次设计任务的得分趋势"，用户可直观判断是否需要触发提示词优化
- 用户可以手动调整某个专家的模型配置（对应 D6 决策的"用户可覆盖"）

### 2.6 元团队评审功能（蓝图应用后回访）

**定位：** 补上"设计-使用-反馈"闭环。用蓝图创建的项目团队运行一段时间后，用户可触发元团队评审，让元团队回头看这个团队设计得好不好，产出评审报告。

**触发入口：** 项目聊天窗顶部栏的"元团队评审"按钮（仅当项目通过蓝图应用创建时显示，即 `blueprint_source` 字段非空）。

**评审输入设计（多维度信息采集）：**

评审启动时，系统自动采集以下信息，组合成评审输入包：

| 维度 | 采集内容 | 采集方式 | 目的 |
|---|---|---|---|
| 原始蓝图 | 当初的 team_blueprint.json 完整内容 | 从 blueprints 库按 blueprint_source 读取 | 让元团队知道"当初的设计意图" |
| 团队配置现状 | 当前项目智能体列表（名称、角色、模板、权限） | 从 conversations.json 读取 | 对比"设计 vs 现状"，发现是否被手动改过 |
| 任务执行统计 | 任务总数、完成数、pending 数、失败数 | 扫描 `.agent/tasks/` 目录的 meta.json | 判断团队是否正常运转 |
| 任务结果摘要 | 最近 N 个任务的 description + status + result_files 文件名 | 扫描 meta.json，不读文件内容 | 了解团队在做什么、做得怎样 |
| 消息交互摘要 | 总消息数、按智能体统计消息数、最近一轮对话的时间 | 扫描 messages.json（按 conversation_id 筛选） | 判断团队活跃度和协作频率 |
| 异常事件记录 | hallucination_warning 次数、工具调用失败次数 | 扫描 messages.json 中的 error 标记 | 发现设计缺陷导致的系统性问题 |
| 用户主观反馈 | 用户填写的一段文字（可选） | 评审对话框中文本框 | 用户的第一手感受，最直接的反馈 |

**不采集的内容（避免 token 爆炸和隐私问题）：**
- 不采集完整聊天记录（只统计摘要）
- 不采集任务 output/ 文件内容（只看文件名和大小）
- 不采集智能体私有工作区内容
- 不采集 deliverables/ 文件内容（只看文件列表）

**评审执行流程：**

1. 用户在项目聊天窗点击"元团队评审"按钮
2. 系统采集评审输入包（上述七个维度）
3. 用户可选择填写主观反馈（可选，可跳过）
4. 系统将评审输入包发送给原班元团队专家（带原始蓝图作为参照）
5. 每个专家独立诊断"这个团队运行中暴露了哪些设计问题"，产出诊断意见
6. 主智能体汇总各专家诊断，产出最终评审报告
7. 评审报告存档到 `meta_team_reviews/{review_id}/report.md`
8. 评审报告同时在项目聊天窗展示给用户

**评审报告内容：**
- 团队设计健康度评分（1-10 分，基于任务完成率、异常事件、协作频率等）
- 设计问题诊断（列出发现的问题，分类：成员配置问题 / 提示词问题 / 协作机制问题 / 权限设计问题）
- 改进建议（针对每个问题，建议怎么改--但不直接改蓝图）
- 是否建议重新设计（是/否，附理由）

**评审报告的后续动作：**
- 评审报告只诊断和建议，不直接改蓝图
- 如果评审建议"重新设计"，用户可在评审报告页面直接点击"启动新设计任务"按钮
- 新设计任务自动带上"上次蓝图 + 评审报告"作为输入，专家能看到"上次设计哪里有问题"
- 产出 v2 蓝图后，用户决定是否应用到原项目（替换团队）或创建新项目

**评审记录管理：**
- 评审记录存储：`backend/meta_team_reviews/{review_id}/`
- 包含文件：`input.json`（评审输入包）、`expert_diagnoses/`（各专家诊断）、`report.md`（最终报告）
- 蓝图库记录该蓝图被评审的次数和最近评审得分，作为蓝图质量参考

**与专家经验积累的联动：**
- 评审报告中的"设计问题诊断"会追加到相关专家的经验记录里
- 专家下次设计任务时，能看到"我之前设计的蓝图在应用后暴露了这些问题"
- 这构成专家成长的另一个输入源（除了同行评审，还有应用反馈）

## 三、工作模式重构方案

### 3.1 现状：流水线模式

```
用户需求 → 需求分析师 → 团队架构师 → 提示词工程师 → 质量评审员 → 蓝图
```

**特点：** 串行、分工明确、每步只做自己的事
**问题：**
- 简单需求过度设计（建三人团队也要走四步）
- 复杂需求无法返工（评审发现问题无法回退）
- 视角单一（每个环节只有一个人在思考，容易盲区）
- 串行等待（四步走完才能拿到结果，体验拖沓）

### 3.2 候选方案对比

本文档提出三种候选工作模式，供讨论评审：

#### 方案 A：保留流水线，增加灵活性（小改）

在现有四步流水线基础上：
- 主智能体可判断需求复杂度，简单需求跳过需求分析
- 评审不通过时支持"返工"指令，打回指定环节
- 保留固定四成员

**优点：** 改动最小，风险最低
**缺点：** 没解决根本问题（视角单一、串行等待）

#### 方案 B：全栈专家并行 + 评审融合（用户提议，中改，已确认采用）

```
用户需求
    ↓
┌──────────────┬──────────────┬──────────────┐
│  专家 A       │  专家 B       │  专家 C       │
│ (模型配置1)   │ (模型配置2)   │ (模型配置3)   │
│ 完整蓝图方案  │ 完整蓝图方案  │ 完整蓝图方案  │
└──────┬───────┴──────┬───────┴──────┬───────┘
       │              │              │
       └──────────────┼──────────────┘
                      ↓
        互相评审打分（每个专家给其他专家打分）
                      ↓
        主智能体融合（第三方立场，重新设计，非拼接）
                      ↓
              最终 team_blueprint.json
              + 融合决策说明
```

**核心思路：**
- 3 个"全栈专家"，每个都独立完成一份完整蓝图方案（从需求理解到团队设计到提示词编写）
- 每个专家配置不同模型（利用模型差异产生多样性，不考虑成本）
- 所有专家并行工作，完全独立，不交流
- 完成后互相评审打分（1-10 分 + 文字意见）
- 主智能体作为第三方融合，综合所有方案 + 打分 + 评审意见，重新设计一份完整蓝图
- 主智能体必须输出"融合决策说明"：为什么采纳各方的哪些优点

**已确认的决策：**

| 项目 | 决策 | 说明 |
|---|---|---|
| 专家数量 | 3 个（默认） | 不考虑成本，3 是多样性和评审复杂度的最佳平衡点；用户可手动调到 4-5 个 |
| 专家交流 | 完全独立，不交流 | 保证方案差异最大化，防止共识错误；独立性价值在于"防止一起偏" |
| 评审方式 | 专家互相打分 | 每个专家给其他专家方案打 1-10 分 + 文字评审意见 |
| 评审标准 | 五项 | 团队结构合理性、提示词质量、协作机制设计、需求匹配度、成本效率 |
| 融合决策权 | 主智能体（第三方） | 不是得分最高者，避免"既当运动员又当裁判"；必须输出融合决策说明 |
| 用户模式选择 | 深度模式 / 快速模式 | 用户启动设计任务时自选，不靠智能体判断复杂度 |
| 模型配置 | 系统自动选差异最大 | 从用户已配置的模型里自动挑选 3 个差异最大的（不同服务商优先）；用户可覆盖；只配 1 个模型时自动退化 |
| 蓝图版本管理 | 任务内保留所有版本，蓝图库存最终版 | 每次迭代产出 `team_blueprint_v{N}.json`；用户满意后点"确认产出"写入蓝图库；蓝图记录来源任务 ID |

**优点：**
- 多视角并行，减少盲区
- 模型多样性带来方案多样性
- 并行执行，总耗时约等于单专家耗时（而非四步串行之和）
- 评审有实质内容（多份方案互相对比，而非自我审查）
- 融合有决策说明，可追溯

**风险与缓解：**
- **风险一：融合可能产生"四不像"。** 缓解：主智能体必须输出完整蓝图（而非拼接），并自检一致性；融合决策说明强制要求解释每个采纳点的理由。
- **风险二：所有专家都误解需求。** 缓解：独立性反而能暴露此问题——评审阶段会发现"所有方案都偏了"，主智能体可提示用户澄清需求。

#### 方案 C：流水线 + 并行混合（中改，未采用）

- 需求分析环节保留（1 人，快速过一遍需求）
- 团队设计 + 提示词工程环节并行化（2-3 人各出完整方案）
- 评审融合环节保留（1 人，负责对比和融合）

```
用户需求 -> 需求分析(1人) -> ┌──────────────┬──────────────┐
                            │  专家 A       │  专家 B       │
                            │ 完整方案      │ 完整方案      │
                            └──────┬───────┴──────┬───────┘
                                   └──────┬───────┘
                                          ↓
                                  评审融合(1人)
                                          ↓
                                  最终蓝图
```

**未采用理由：** 保留了流水线的串行包袱，不够干净。用户明确倾向纯粹的"全栈并行 + 评审"模式。

### 3.3 方案对比矩阵

| 维度 | 方案 A（流水线优化） | 方案 B（全栈并行，已采用） | 方案 C（混合，未采用） |
|---|---|---|---|
| 改动幅度 | 小 | 中 | 中 |
| Token 成本 | 1x | 3x（不考虑成本） | 1.5-2x |
| 方案多样性 | 低 | 高 | 中 |
| 总耗时 | 4 步串行 | 1 步并行 + 评审 | 1 步 + 并行 + 评审 |
| 简单需求处理 | 可跳步 | 用户选快速模式（1 专家） | 可跳过需求分析 |
| 返工能力 | 支持 | 重新跑一轮 | 支持 |
| 实现复杂度 | 低 | 中 | 中高 |

### 3.4 已确认采用方案 B

**用户已确认采用方案 B（全栈专家并行 + 评审融合），核心决策见 3.2 节方案 B 的"已确认的决策"表格。**

**方案 B 的完整数据结构：**

```
meta_team_tasks/{task_id}/
├── meta.json                  # 任务元数据（状态、模式、专家配置、创建时间）
├── requirement.md             # 需求记录（用户输入 + 澄清对话）
├── proposals/                 # 各专家完整方案
│   ├── expert_1.md            # 完整蓝图方案（含团队结构、提示词、协作机制）
│   ├── expert_2.md
│   └── expert_3.md
├── reviews/                   # 评审记录（每个专家对其他方案的打分和意见）
│   ├── expert_1_review.md
│   ├── expert_2_review.md
│   └── expert_3_review.md
├── team_blueprint_v1.json     # 最终蓝图（每次迭代一个版本）
├── team_blueprint_v2.json
└── fusion_decision.md         # 主智能体的融合决策说明
```

**方案 B 的完整流程：**

1. 用户启动设计任务，选择"深度模式"或"快速模式"
2. 深度模式：系统自动选 3 个差异最大的模型配置专家（用户可覆盖）；快速模式：1 个专家
3. 专家并行工作，各出完整蓝图方案（含团队结构、成员提示词、协作机制）
4. 专家互相评审：每个专家给其他专家方案打 1-10 分 + 文字意见（按五项标准）
5. 主智能体融合：综合所有方案 + 打分 + 评审意见，重新设计一份完整蓝图（非拼接）
6. 主智能体输出融合决策说明：解释每个采纳点的理由
7. 用户审阅最终蓝图：满意则"确认产出"入库；不满意则"重跑"迭代

### 3.5 评审标准（五项）

| 评审项 | 说明 | 打分维度 |
|---|---|---|
| 团队结构合理性 | 成员配置是否合理、分工是否清晰、有无冗余或缺失 | 1-10 分 |
| 提示词质量 | 清晰度、可执行性、角色边界是否明确 | 1-10 分 |
| 协作机制设计 | 谁向谁汇报、并行还是串行、协作模式是否匹配任务 | 1-10 分 |
| 需求匹配度 | 团队设计是否覆盖了需求的全部要点、有无偏离 | 1-10 分 |
| 成本效率 | 有无过度设计、成员数量是否最优 | 1-10 分 |

## 六、结论

本方案提出两个层面的重构，并配套评审闭环和专家进化机制：

1. **架构层面：** 元团队从"项目类型"重构为"常驻功能模块"，解决归类错误、一次性浪费、入口错位三个结构性问题。已有元团队项目全部废弃删除。元团队专家全局常驻，不是每任务创建。

2. **工作模式层面：** 从"串行流水线"重构为"并行全栈专家 + 评审融合"（方案 B），解决视角单一、串行等待、刚性流程三个体验问题。3 个专家独立并行出完整方案，互相评审打分，主智能体第三方融合并输出决策说明。

3. **评审闭环：** 蓝图应用后的项目可触发元团队回访评审，采集多维度运行数据（原始蓝图、任务统计、消息摘要、异常事件、用户反馈），专家诊断设计问题，产出评审报告。评审报告可触发新一轮设计任务，形成"设计-使用-反馈-改进设计"闭环。

4. **专家进化：** 两层成长机制。第一层经验自动积累（每次任务后自动记录得分和反馈，专家有记忆）。第二层提示词版本升级（用户主导，外部模型提建议，用户确认后升级，不自动修改）。

分四阶段实施（常驻化 -> 工作模式 -> 评审闭环 -> 优化迭代），各阶段可独立验证，前阶段失败不影响后续。

**本方案已经用户讨论确认 14 项核心决策（见第五节 D1-D14），可进入实施规划阶段。**
