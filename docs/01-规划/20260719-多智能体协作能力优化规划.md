# 多智能体协作能力优化规划

## 一、问题背景

端到端 LLM 测试暴露出一个共性问题：主智能体在多轮对话后出现"偷懒"现象。

测试数据（6 个场景，优化前通过 2 个，加入 CORE_RULES 后通过 3 个）：

| 场景 | 优先级 | 优化前 | 优化后 | 现象 |
|------|--------|--------|--------|------|
| TC-E2E-01 小说创作完整流程 | P0 | ✓ | ✓ | 单轮明确指令，表现稳定 |
| TC-E2E-06 归档流程含校验证据 | P0 | ✗ | ✓ | CORE_RULES 生效，归档动作正确执行 |
| TC-E2E-08 全流程无幻觉 | P0 | ✓ | ✓ | 防幻觉机制有效 |
| TC-E2E-02 小说创作+审阅协作 | P1 | ✗ | ✗ | 37 token、0 工具调用 |
| TC-E2E-04 多人并行写多章 | P1 | ✗ | ✗ | 40 token、0 工具调用 |
| TC-E2E-05 协调者调度L2执行者 | P1 | ✗ | ✗ | 23 token、0 工具调用 |

3 个 P1 场景的失败特征高度一致：主智能体回复了约 15-30 个汉字（"好的我来安排"之类），但没有调用任何工具。对执行型智能体而言，不调工具等于没干活。

### 根本原因

纯提示词方案（Layer 1）对单轮明确指令有效，但对多轮延续调度效果有限。原因有三：

1. **历史消息累积**：`build_conversation_history` 的 `max_history=20` 会保留前面所有对话。前 3 轮包含大量工具调用总结，这些文字占据大部分上下文空间，规则提示词被淹没。

2. **无系统级工作流编排**：主智能体没有显式的待办清单（TODO List），所有待办事项都装在对话历史里。聊到第 4 轮，前面说的"待办"就被新对话淹没。

3. **子智能体回复全量回流**：子智能体执行完后的完整回复（可能 2000+ 字）会全部写入主智能体的对话历史，导致上下文膨胀，规则约束力衰减。

### 与元宝等 APP 的差异

元宝等问答型智能体回复文字即算完成，不存在"答应了但没做"的问题。P005 是执行型智能体，判断"是否真正执行"的标准是工具调用，不是文字回复。这是所有 AI Agent 框架面临的共性挑战。

## 二、目标与范围

### 目标

将端到端 LLM 测试通过率从 3/6 提升到 5/6 以上，重点解决多轮调度场景的主智能体偷懒问题。

### 范围

本规划聚焦三个核心机制，按优先级排序：

| 编号 | 机制 | 优先级 | 预计代码量 | 预期效果 |
|------|------|--------|-----------|---------|
| M1 | 子代理结果摘要机制 | P0 | ~100 行 | 直接解决多轮偷懒，立竿见影 |
| M2 | TODO 清单系统 | P0 | ~400 行 | 治本，主智能体自带"备忘录" |
| M3 | 上下文窗口 token 管理 | P1 | ~150 行 | 防止长对话崩溃 |

### 不做

- 工具结果智能截断（P2，后续迭代）
- 按任务类型动态授权（P2，后续迭代）
- 错误自动重试（P3，后续迭代）
- 计划模式（P1，待 M1-M3 完成后评估）

## 三、设计方案

### M1：子代理结果摘要机制

#### 设计思路

借鉴 Trae 的子代理（Subagent）模式：子智能体执行完后，只把"结果摘要"写入主智能体历史，详细过程存到任务文件夹。这样主智能体的上下文永远保持精简。

#### 数据流对比

**当前流程**：
```
子智能体回复（2000字小说章节）
  → 全量写入 messages.json
  → 全量进入主智能体 build_conversation_history
  → 主智能体上下文膨胀
```

**优化后流程**：
```
子智能体回复（2000字小说章节）
  → 全量写入 messages.json（保留完整记录）
  → 生成摘要（≤200字）写入主智能体历史
  → 详细内容存到 .agent/tasks/{task_id}/output/
  → 主智能体上下文保持精简
```

#### 实现细节

        role = "user" if msg.get("role") == "user" else "assistant"
        content = msg.get("content", "")

### M2：TODO 清单系统

#### 设计思路

借鉴 Trae 的 `TodoWrite` 工具 + 系统层强制。主智能体接到复杂任务后，先拆解成步骤，每步显式标记状态。系统在每轮回复前检查 TODO 清单，未完成时自动提醒。

#### 数据结构

TODO 清单存储在 `.agent/todos/{todo_id}.json`：

```json
{
  "todo_id": "todo_abc123",
  "conversation_id": "proj_xxx",
  "created_at": "2026-07-19 14:00:00",
  "updated_at": "2026-07-19 14:05:00",
  "title": "写武侠小说",
  "items": [
    {
      "index": 0,
      "content": "创建创作者（creator 模板）",
      "status": "completed",
      "updated_at": "2026-07-19 14:01:00"
    },
    {
      "index": 1,
      "content": "创建审阅者（reviewer 模板）",
      "status": "pending",
      "updated_at": "2026-07-19 14:00:00"
    }
  ],
  "status": "in_progress"
}
```

状态取值：`pending`（待处理）、`in_progress`（进行中）、`completed`（已完成）。

#### 新增工具

**工具 1：`create_todo_list`**

```python
def create_todo_list(title: str, items: list, conv_id: str, root_path: str) -> str:
    """创建任务清单"""
    todo_id = f"todo_{uuid.uuid4().hex[:8]}"
    todo = {
        "todo_id": todo_id,
        "conversation_id": conv_id,
        "created_at": time.strftime("%Y-%m-%d %H:%M:%S"),
        "title": title,
        "items": [
            {"index": i, "content": item, "status": "pending", 
             "updated_at": time.strftime("%Y-%m-%d %H:%M:%S")}
            for i, item in enumerate(items)
        ],
        "status": "in_progress"
    }
    # 保存到 .agent/todos/{todo_id}.json
    todos_dir = os.path.join(root_path, ".agent", "todos")
    os.makedirs(todos_dir, exist_ok=True)
    with open(os.path.join(todos_dir, f"{todo_id}.json"), "w", encoding="utf-8") as f:
        json.dump(todo, f, ensure_ascii=False, indent=2)
    return f"✅ 已创建 TODO 清单（{todo_id}），共 {len(items)} 项：\n" + \
           "\n".join([f"  {i+1}. [pending] {item}" for i, item in enumerate(items)])
```

**工具 2：`update_todo_status`**

```python
def update_todo_status(todo_id: str, item_index: int, status: str, root_path: str) -> str:
    """更新 TODO 项状态"""
    todo_path = os.path.join(root_path, ".agent", "todos", f"{todo_id}.json")
    if not os.path.exists(todo_path):
        return f"❌ 找不到 TODO 清单：{todo_id}"
    
    with open(todo_path, "r", encoding="utf-8") as f:
        todo = json.load(f)
    
    if item_index < 0 or item_index >= len(todo["items"]):
        return f"❌ 项索引超出范围：{item_index}（共 {len(todo['items'])} 项）"
    
    if status not in ("pending", "in_progress", "completed"):
        return f"❌ 无效状态：{status}（可选：pending/in_progress/completed）"
    
    todo["items"][item_index]["status"] = status
    todo["items"][item_index]["updated_at"] = time.strftime("%Y-%m-%d %H:%M:%S")
    todo["updated_at"] = time.strftime("%Y-%m-%d %H:%M:%S")
    
    # 检查是否全部完成
    if all(item["status"] == "completed" for item in todo["items"]):
        todo["status"] = "completed"
    else:
        todo["status"] = "in_progress"
    
    with open(todo_path, "w", encoding="utf-8") as f:
        json.dump(todo, f, ensure_ascii=False, indent=2)
    
    return f"✅ 已更新：第 {item_index+1} 项 → {status}"
```

`check_pending_todos` 函数：

```python
def check_pending_todos(conv_id: str, root_path: str) -> str:
    """检查未完成的 TODO 清单，返回提醒文本"""
    todos_dir = os.path.join(root_path, ".agent", "todos")
    if not os.path.exists(todos_dir):
        return ""
    
    reminders = []
    for fname in os.listdir(todos_dir):
        if not fname.endswith(".json"):
            continue
        with open(os.path.join(todos_dir, fname), "r", encoding="utf-8") as f:
            todo = json.load(f)
        if todo.get("conversation_id") != conv_id:
            continue
        if todo.get("status") != "in_progress":
            continue
        
        pending_items = [item for item in todo["items"] if item["status"] != "completed"]
        if pending_items:
            reminders.append(f"📋 TODO 清单「{todo['title']}」还有 {len(pending_items)} 项未完成：")
            for item in pending_items:
                reminders.append(f"  - [{item['status']}] {item['content']}")
            reminders.append("请继续推进，或明确说明为何停止。")
    
    return "\n".join(reminders) if reminders else ""
```

### M3：上下文窗口 token 管理

#### 设计思路

当前 `max_history=20` 是按消息条数截断，不看 token。如果每条消息很长（比如子智能体写了 2000 字），20 条可能就超限；如果都很短，20 条又浪费上下文空间。

优化为按 token 数智能截断，优先保留最近的消息和系统提示词。

#### 实现方案

**新增 token 计数函数**：

```python
def estimate_tokens(text: str) -> int:
    """粗略估算 token 数（中文约 1.5 字/token，英文约 4 字符/token）"""
    chinese_chars = sum(1 for c in text if '\u4e00' <= c <= '\u9fff')
    other_chars = len(text) - chinese_chars
    return int(chinese_chars * 1.5 + other_chars / 4)

def count_messages_tokens(messages: list) -> int:
    """计算消息列表的总 token 数"""
    total = 0
    for msg in messages:
        total += estimate_tokens(msg.get("content", ""))
        total += 4  # 每条消息的元数据开销
    return total
```

**改造 `build_conversation_history`**：

```python
def build_conversation_history(conv_id, system_prompt, current_user_msg=None,
                                max_history=20, root_path="",
                                target_agent_id="", for_main_agent=False,
                                max_tokens=30000):
    # ... 构建 messages 列表（原有逻辑）...
    
    # 新增：按 token 数智能截断
    # 始终保留：system 消息 + 最近 3 轮（6 条）+ 当前用户消息
    # 中间历史按 token 预算动态保留
    if count_messages_tokens(messages) > max_tokens:
        system_msg = messages[0]
        current_msg = messages[-1] if current_user_msg else None
        recent_msgs = messages[-7:-1] if current_user_msg else messages[-6:]
        
        # 从最早的历史开始删除，直到 token 数达标
        middle_msgs = messages[1:-7] if current_user_msg else messages[1:-6]
        while middle_msgs and count_messages_tokens([system_msg] + middle_msgs + recent_msgs + ([current_msg] if current_msg else [])) > max_tokens:
            middle_msgs.pop(0)
        
        messages = [system_msg] + middle_msgs + recent_msgs
        if current_msg:
            messages.append(current_msg)
    
    return messages
```

#### 验证标准

- 上下文 token 数不超过 30000（可配置）
- 始终保留 system 消息和最近 3 轮对话
- 长对话不再因上下文超限而报错

## 六、风险与应对

### 风险 1：M1 摘要机制导致主智能体丢失关键信息

**应对**：摘要保留前 200 字 + 字数提示，主智能体需要细节时可主动调用 `read_file` 读取任务文件夹的完整产出。在提示词中明确告知"完整内容在 .agent/tasks/{task_id}/output/"。

### 风险 2：M2 TODO 清单被主智能体滥用

**应对**：在提示词中明确"何时必须用"和"何时不用"，避免简单任务也创建清单。设置清单数量上限（单对话最多 5 个），超过则拒绝创建。

### 风险 3：M3 token 估算不准

**应对**：采用粗略估算（中文 1.5 字/token，英文 4 字符/token），保守设置 `max_tokens=30000`（留足余量）。如果实际超限，再调整估算系数。

### 风险 4：系统层强制导致主智能体行为僵化

**应对**：TODO 提醒采用"建议"语气而非"命令"语气，主智能体仍可选择不执行（比如用户明确说"先停一下"）。提醒文本末尾加"请继续推进，或明确说明为何停止"。
